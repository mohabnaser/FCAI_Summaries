01

What is a State Diagram?

3 أسامي لنفس الـ diagram: State Diagram = State Machine Diagram = Statechart.

📖 Definition

State diagram بيوصف الـ behavior بتاع:

  • A whole system, or
  • Some part of a system, or
  • An individual object (الأكتر شيوعاً).

🔑 الفكرة الأساسية

  • في أي point in time، النظام/الـ object في state معين.
  • كونه في state يعني أنه هيتصرف بشكل محدد ردًا على الـ events.
  • بعض الـ events بتسبب change of state (transition).
  • في الـ state الجديد، الـ behavior بيكون مختلف.
🧮 الـ Formal View

State diagram = directed graph:

  • Nodes = states (الحالات)
  • Arcs (edges) = transitions (الانتقالات)
💡 الفرق المهم عن الـ Sequence diagramالـ Sequence = كيف objects كتيرة بتتفاعل لتنفيذ use case. الـ State = حياة object واحد بمرور الزمن. الاتنين dynamic بس بزاوية مختلفة.

💡 ايه يعني "State"؟

03

Transitions & Events

Transition = change of state. Event = اللي بيسبب الـ transition.

➡️ Transition

الـ Transition بيمثل تغيير في الـ state ردًا على event. الـ label على الـ transition بيوصف:

  1. الـ Event (الأهم).
  2. [Guard] (شرط اختياري).
  3. / Action (action اختياري).
event(parameters) [guard] / action

مثال: requestToRegister(student) [classSize < max] / createRegistration

📞 Mini Example — Telephone
idle
off-hook
dial tone
ⓘ off-hook = event · idle, dial tone = states · ━━▶ = transition
04

3 Event Types ⭐

External · Internal · Temporal. الـ doctor بيحب يسأل عن الفرق.

🌐 External Event

= System Event

سببه حاجة برّه الـ system boundary.

مثال: Cashier بيضغط "enter item" button على POS terminal.

🔧 Internal Event

سببه حاجة جوّه الـ system boundary.

Operation invoked via message من object تاني داخلي.

مثال: Object بيـ call method على object تاني — ينتج عنه event.

⏰ Temporal Event

سببه occurrence of a specific date/time or passage of time.

الـ syntax: after(30s) أو at(midnight).

مثال: Red light → Green light بعد 30s.

📌 ملحوظةالـ Temporal events بتكتب بـ after(...) للـ duration أو at(...) لـ specific time. شائعة في الـ state diagrams.
05

Activities vs Actions ⭐

الفرق المهم: Action = instantaneous. Activity = takes time.

⚡ Action

Takes NO noticeable time (instantaneous)

الـ Action بيحصل في 3 حالات:

  • On transition — مع الانتقال نفسه
  • On entry — لما ندخل state معين
  • On exit — لما نخرج من state معين

أمثلة: readCard, ejectCard, startMotor, stopMotor

Syntax: entry / actionName
Or: exit / actionName

🏃 Activity

Takes a period of time (ongoing)

الـ Activity بيحصل while in state:

  • بياخد فترة زمنية ملحوظة
  • ممكن الـ system يخرج من الـ state لما يخلص الـ activity (completion)
  • أو يخرج لو event آخر يقطعها (interruption — early exit)

أمثلة: play music, play dial tone, download file

Syntax: do / activityName

📝 Concrete Example — ATM
Serving Customer entry / readCard exit / ejectCard do / process transaction
  • الـ entry / readCard = action بتحصل مرة واحدة لما ندخل الـ state.
  • الـ do / process transaction = activity بتشتغل طول ما إحنا في الـ state.
  • الـ exit / ejectCard = action بتحصل لما نخرج من الـ state.
06

Guards & Conditional Transitions

Guard = شرط بوليان لازم يكون true عشان الـ transition تحصل.

🔒 Guard Syntax
event [boolean expression] / action

الـ guard بين [ ]. لو الـ event حصل بس الـ guard مش true، الـ transition ما بتحصلش.

مثال: Course Registration

OpenNotEnoughStudents
[classSize ≥ minimum]
OpenEnoughStudents
لو classSize ≥ minimum = true، الـ transition بتحصل. لو false، الـ state ما يتغيرش.
⚙️ Self-Transitions with Guards

State diagram ممكن يكون فيه transition من state لنفسها. مفيدة لما لازم عمل action بدون تغيير state.

مثال: requestToRegister(student) / createRegistration — self-transition بـ action. الـ state يفضل OpenNotEnoughStudents بس createRegistration بيتعمل.

07

Nested / Composite States

State جوّه state. مفيد لتنظيم complexity في الـ diagram.

🪆 Composite State (Substate)

الـ Composite state بيحتوي على substates داخله. شكلهم مستطيل أكبر بـ states أصغر جوّاه.

الفوائد:

  • تنظيم — تخفّض عدد transitions ظاهرة في الـ diagram.
  • الـ outer state بتنطبق على كل الـ substates (مثلاً exit من any substate = exit from composite).
  • Easier reading and maintenance.

مثال: ATM "Serving Customer" composite

Serving Customer
entry / readCard · exit / ejectCard
→ Authenticate → Select Trans. → Transaction →

الـ outer transitions (e.g., cancel) بتشتغل من أي substate داخل Serving Customer. مفيش حاجة نضيف cancel arrow من كل substate لوحده.

💡 متى نستخدم composite states؟لما عندك مجموعة من الـ states ليها common behavior (مثلاً كلهم ممكن يـ cancel) أو بتنتمي لنفس الـ logical phase (مثل phases of serving customer).

🏧 ATM (Composite States)

أكثر example معقد. فيه composite state للـ Serving Customer.

  • Off → Self Test [turn on / startup]
  • Self Test → Idle (success)
  • Idle → Serving Customer [cardInserted]
  • Inside Serving Customer (composite): Authenticate → Selecting → Transaction → end
  • Serving Customer → Idle [cancel]
  • Idle → Maintenance [service]
  • Maintenance → Out of Service [failure]
  • Out of Service → Off [turn off / shutDown]
💡 Composite Patternالـ Serving Customer composite بتحتوي substates للـ authentication flow. لو حصل cancel من أي substate، الـ ATM يرجع لـ Idle. ده استخدام نظيف للـ composite.

Practice Quiz — Lecture 8

8 أسئلة على State Diagrams. Question 2 من كل امتحان فيه 2-4 marks على state diagram.

Question 1 Practice — concept
A state diagram describes the behavior of:
A Multiple interacting objects over time
B An individual object (or a system) over time
C Static structure of the system
D Use case relationships
✅ الإجابة الصح: B — Individual object / system State diagram = behavior of one object (أو one system as a whole) عبر الزمن.
A — ده الـ Sequence diagram (multiple objects). · C — ده الـ Class diagram. · D — ده الـ Use Case diagram.
Question 2 Practice — events
A traffic light changes after 30 seconds. What type of event is this?
A External Event
B Internal Event
C Temporal Event
D Composite Event
✅ الإجابة الصح: C — Temporal Event after(30s) = passage of time = Temporal event.
A (External) — لما الـ trigger من برّه (مثل user click). · B (Internal) — من object داخلي. · D (Composite) — مفيش "composite event" — الـ composite للـ states.
Question 3 Practice — action vs activity
Inside a state box, you see do / play music. This is:
A An activity (takes time)
B An entry action
C An exit action
D A guard condition
✅ الإجابة الصح: A — Activity الـ keyword do / يدل على activity (ongoing). الـ "play music" بياخد فترة.
B — Entry action syntax: entry / actionName. · C — Exit action syntax: exit / actionName. · D — Guards بين [ ].
Question 4 Practice — transition syntax
Given the transition label buy(item) [funds >= price] / debitAccount, what does [funds >= price] represent?
A An event
B An action
C A state name
D A guard condition
✅ الإجابة الصح: D — Guard اللي بين [ ] = guard (boolean expression). لو true → transition تحصل.
A (event) — هو buy(item) قبل الـ guard. · B (action) — debitAccount بعد الـ /. · C — مش state name.
Question 5 Practice — symbols
What represents the start state in a state diagram?
A A circle with a ring around it
B A solid black circle
C A rounded rectangle with "start" inside
D A diamond shape
✅ الإجابة الصح: B — Solid black circle الـ Initial state = filled black circle.
A — هو الـ end state (black circle محاطة بـ ring). · C — Regular states are rounded rectangles مع اسمهم، مش "start". · D — Diamond ده في activity diagrams (decision).
Question 6 Practice — composite
An ATM diagram shows a composite state called ServingCustomer with substates inside, and a cancel transition from ServingCustomer to Idle. What does this mean?
A Cancel only works in the first substate
B Cancel works only when there are no substates
C Cancel works from any substate inside ServingCustomer
D Cancel is forbidden inside the composite
✅ الإجابة الصح: C — From any substate الـ Transition من الـ outer boundary للـ composite بتشتغل من أي substate جوّاها. ده الفائدة الأساسية للـ composite states.
A, B, D — كلهم بيناقضوا فكرة الـ composite states.
Question 7 Practice — when used
When is a state diagram the most useful kind of UML diagram?
A When modeling many objects interacting
B When modeling code structure
C When modeling an object/system with distinct states and transitions between them
D When listing all user actions
✅ الإجابة الصح: C State diagram يبقى مفيد لما عندك object/system في حالات مختلفة (modes) ـ مع events بتسبب transitions. مثل: ATM, Order lifecycle, Door, Game.
A — ده sequence diagram. · B — class diagram. · D — use case diagram.
Question 8 Practice — comprehensive
An Order in an e-commerce site has states: Placed → Paid → Shipped → Delivered. The customer can Cancel from any state before Shipped. Which UML element best represents this?
A Cancel transitions from each state separately to a Cancelled state
B A composite state grouping Placed+Paid, with a single Cancel transition from the composite to Cancelled
C A guard condition on the Shipped state
D Mark Cancelled as the initial state
✅ الإجابة الصح: B — Composite state الطريقة الأنظف: composite state يشمل Placed و Paid، والـ Cancel من الـ composite boundary. ده بيقلل عدد arrows وبيعكس الـ logical grouping (orders اللي ممكن تتلغى).
A — يشتغل لكن بيـ clutter الـ diagram. الـ composite أنظف. · C — Guard على state غير منطقي هنا. · D — Cancelled لازم يكون final state، مش initial.

🚨 Common Mistakes

1. خلط Action و ActivityAction = instantaneous (entry/exit/on-transition). Activity = takes time (do /). الـ "do /" بيدل دايماً على activity.
2. نسيان Initial State أو Final Stateأي state diagram لازم يبدأ بـ ● initial state. لو فيه نهاية للـ behavior، حط ⊙ final state.
3. نسيان labels على الـ transitionsكل transition لازم لها label (event). بدون event، الـ transition ما تحصلش بشكل مفهوم.
4. خلط State Diagram و Sequence DiagramState = behavior of ONE object over time. Sequence = interaction between MANY objects. لو سؤال عن "كيف objects بتتكلم"، ده Sequence. لو "كيف status بيتغير"، ده State.
5. عدم استخدام composite statesلما عندك transition مشترك من كذا state (مثل cancel من أي state)، استخدم composite. غير كده الـ diagram هيبقى cluttered بأسهم كتيرة.
