What is UML?

الـ Unified Modeling Language — لغة رسومية معيارية لنمذجة الـ OO software.

📖 الخلفية
  • طوّروها Rumbaugh, Booch, Jacobson (الـ "Three Amigos").
  • أسّسوا Rational Software Corporation، اللي اشترتها IBM بعد كده.
  • في 1997، الـ Object Management Group (OMG) بدأت standardization process.
📚 أنواع UML Diagrams

🏗️ Static View

  • Class diagrams — describe classes & relationships — بتوصف الـ classes والعلاقات بينها.
  • Component diagrams
  • Deployment diagrams

🎬 Dynamic View

  • Sequence diagrams — object interactions — التفاعلات بين الـ objects.
  • Communication diagrams
  • State diagrams (internal behavior) — السلوك الداخلي للـ object.
  • Activity diagrams
💡 هذا الـ lectureهنركز على Class Diagrams (static structure). الـ Sequence/State diagrams في lectures جاية.

🔄 Class Diagrams في الـ lifecycle

الـ Class diagrams بتختلف حسب المرحلة:

  • Domain Analysis — مفاهيم الـ business عالية المستوى (مفيش methods، attributes بس).
  • Requirements Specification — بتضيف الـ operations اللي ظهرت من الـ use cases.
  • Implementation — blueprint مباشر للكود مع visibility و types و library classes.

بتتطور iteratively مع الوقت.

Associations & Multiplicity

الـ Associations بتربط classes ببعضها. الـ Multiplicity بتقول كم instance من كل side.

🔢 Multiplicity Reference
1exactly one
*zero or many
0..1zero or one (optional)
1..*at least one
3..8between 3 and 8 inclusive
0, 3..8either zero, or between 3 and 8
🔗 Many-to-One Example
Employee
* ━━━ worksFor ━━━ 1
Company
  • A company has many employees (*).
  • An employee works only for 1 company.
  • A company can have zero employees (e.g. shell company).
  • An employee must work for a company (multiplicity 1, not 0..1).
🔗 Many-to-Many Example
Assistant
* ━━━ supervisor ━━━ 1..*
Manager
  • An assistant can work for many managers (1..*) — الـ assistant ممكن يشتغل لـ عدة managers.
  • A manager can have many (or 0) assistants — الـ manager ممكن يكون عنده كذا assistant أو ولا حد.
  • Question: is it possible for assistant to have zero managers temporarily?

لو الـ 1:1 إلزامية دايماً، اعملها attribute داخل الـ class الواحدة، مش relation منفصلة.

مثال: grade للـ student في course. مش property للـ student (عنده grades كتيرة)، ومش property للـ course (فيه students كتير). هو property للـ registration.

Class بتربط نفسها بـ نفسها. مثال: Course:

Generalization (Inheritance)

Superclass → Subclasses. Open triangle arrow point لـ superclass.

🌳 Basic Generalization
Animal
△
discriminator: habitat
AquaticAnimal

Subclasses بترث attributes و operations من الـ superclass.

Generalization Set = grouping of subclasses based on a discriminator (criterion of specialization).

🔀 Multiple Discriminators

Animals يتقسموا بأكتر من معيار: habitat (aquatic/land) و typeOfFood (carnivore/herbivore). حلين:

Option 1: Higher-level Generalization

Animal → AquaticAnimal → AquaticCarnivore + AquaticHerbivore (وكذلك Land).

⚠️ بيعمل classes كتيرة (4 leaves).

Option 2: Multiple Inheritance

AquaticCarnivore inherits from both AquaticAnimal and Carnivore.

⚠️ مش كل اللغات بتدعمها (Java لا، C++ يدعمها).

🚫 Avoid Instances Changing Class

لو الـ instance ممكن يتغير من type لـ type تاني، لا تستخدم inheritance. استخدم attribute بدالها.

06

Aggregation vs Composition ⭐

"Part-whole" relationships. الفرق بينهم: هل الـ part تقدر تعيش بدون الـ whole؟

💠 Aggregation (Open Diamond ◇)

"part-whole" بس الـ parts تقدر تعيش مستقلة عن الـ whole.

Vehicle
1 ◇━━ *
VehiclePart
  • Vehicle parts ممكن تنبع من cars مختلفة (reused).
  • لو الـ Vehicle اتدمر، الـ parts ممكن تستخدم في Vehicle تاني.

✅ When to use Aggregation:

  • You can say "the parts are part of the aggregate" أو "the aggregate is composed of the parts".
  • When something owns/controls the aggregate, then it also owns/controls the parts — اللي بيمتلك الـ aggregate بيمتلك الأجزاء كمان.
♦️ Composition (Filled Diamond ◆)

Strong aggregation. لو الـ aggregate اتدمر، الـ parts اتدمرت معاه. الـ part ما تقدرش تعيش بدون الـ whole.

Building
1 ◆━━ *
Room
  • الـ Rooms ما تقدرش توجد بدون الـ Building.
  • لو الـ Building اتهد، الـ Rooms راحت معاه.
⚖️ Aggregation vs Composition — جدول مقارنة
AspectAggregation ◇Composition ◆
SymbolOpen (hollow) diamondFilled (solid) diamond
Part lifecycleIndependentTied to whole
If whole destroyedParts surviveParts destroyed
Part can be reusedYes (in other wholes)No (one whole only)
ExampleVehicle - VehiclePart · Country - RegionBuilding - Room · Book - Chapter (if Chapter can't exist alone)

🔄 Propagation

Mechanism: operation في الـ aggregate بتنفذ بأن الـ aggregate يـ perform نفس الـ operation على parts بتاعته.

مثال: Polygon.draw() بينفذ بأن الـ Polygon يطلب من كل LineSegment أن يـ draw().

وعكسياً: properties للـ parts بتنتشر للـ aggregate.

08

Full Example: Problem Report Tool

CASE tool لتخزين وتتبع problem reports. مثال شامل لكل ال relations.

📋 The Requirements
  1. Employees are either developers or managers (generalization).
  2. Each developer has a manager (1 manager per developer) — كل developer ليه manager واحد.
  3. Project assigned a manager + a number of developers — كل project ليه manager وعدد من الـ developers.
  4. Developer/manager may be assigned to multiple projects or to none (0..*) — الـ developer/manager ممكن يكون على projects كتير أو ولا واحد.
  5. Project consists of artifacts (domain analysis, SRS, code, test cases...) — aggregation.
  6. Problem reports can be made on any artifact — تقدر تعمل problem report على أي artifact.
  7. Each report contains: problem description and status.
  8. Each problem can be assigned to someone by the project manager — الـ project manager ممكن يـعيّن المشكلة لحد معيّن.
  9. Report has history entries (action/update logs) — aggregation.
  10. A special kind of reports: Code/Bug Reports with extra info (location, circumstances, failed tests) — generalization.
📐 The Class Diagram (high-level)
        ┌──────────┐
        │ Employee │ (name: String)
        └────△─────┘
             │
       ┌─────┴──────────┐
   ┌───┴───┐      ┌─────┴────┐
   │Manager│      │Developer │
   └───┬───┘      └────┬─────┘
       │              │   * managed by 1
       │ 1            └───↑─────────────┐
       │  responsibleFor  │            Manager
       │  0..*            │ * assigned 0..*
   ┌───↓──────────┐  ┌──↓──────────┐
   │ Problem Rep. │  │  Project    │ (name)
   │ + status     │  └──◇──────────┘  ← aggregation
   │ + desc       │     │ 1
   │              │     │ 0..*
   └──△───────────┘  ┌──↓─────────┐
      │              │  Artifact  │ ── associated 0..n ──→ Problem Report (label: "About")
   ┌──┴──────────┐   │ +name      │
   │ CodeBugRep. │   │ +status    │
   └─────────────┘   └────────────┘

   Problem Report ◇──── 0..n ───→ History Entry (when: Date, whatDone: String)
                  "History Log"
💡 الـ relations الـ 5 الموجودة في المثال
1. Generalization — Manager & Developer extend Employee. CodeBugReport extends Problem Report.
2. Associations — Developer managedBy Manager, Employee assignedTo Project, Manager responsibleFor Problem Report, Artifact "About" Problem Report.
3. Aggregation — Project ◇ Artifact, Problem Report ◇ History Entry.
4. Multiplicities — كلهم: 1, *, 0..*, 1..*, 0..n.
5. Labels — "Managed By", "Assigned To", "Responsible For", "About", "History Log".

13 Slide Exercises — Practice

التمارين اللي د. الرملي عاملها في السلايدز. لازم تحلهم كلهم — كل واحد بيختبر concept مختلف.

Ex 1 Shape Class & Square — What relation?

Shape ← Triangle, Square (open-triangle generalization arrow).

الإجابة: Generalization (Inheritance) — Square inherits from Shape. Triangle inherits from Shape.
Ex 2-4 Account ↔ Client (Bank example)

Class diagram: Account [1] ━━ [0..n] Client +owner

  • Ex 2: What relation? → Simple Association with multiplicity.
  • Ex 3: What's true?
    • A client can own multiple accounts? True (0..n on client side from account's perspective is the inverse — actually account is 1 to many clients here meaning 1 account has 0..n owners; the question is open to interpretation, but conventional answer follows the diagram).
    • An account can be used to access its client? True (bidirectional by default).
    • A client can be used to access its account? True.
  • Ex 4: What's true?
    • An account can have multiple owners? True (0..n on client side).
    • No account can exist without a client? Depends — if 0..n includes 0, account can exist without clients.
    • A client must have an account? True if multiplicity is 1 on account side.
Ex 5 Building ◆━ Room (Composition)
  • It shows a composition relation? ✓ True (filled diamond).
  • A building may have no rooms? ✗ False if multiplicity is 1..* (depends on diagram).
  • A room may exist without a building? ✗ False — composition means parts die with whole.
Ex 6 Employee ━ Address (Plain Association, 1:1)
  • It shows a composition relation? ✗ False (no diamond shown).
  • An employee may have no address? ✗ False if multiplicity is 1 (mandatory).
  • An address may exist without an employee? ✓ True if it's a plain association.
Ex 7 Employee → Manager/Developer, Project assignment

Part 1: Manager [1] managedBy [0..*] Developer (subclasses of Employee).

  • Employee can be Manager and Developer at the same time? ✗ False (in single inheritance, subclasses are disjoint).
  • Manager may have no developers assigned? ✓ True (0..* includes 0).
  • Developer has one and only one manager? ✓ True (multiplicity 1).

Part 2: Employee [1..*] assignedTo [0..*] Project.

  • Project has at least one employee? ✓ True (1..*).
  • Every developer must be assigned to a project? ✗ False (0..* on project side means dev can have zero projects).
  • Manager may have no projects? ✓ True.
Ex 8 Course (Reflexive: successor, prerequisite, mutuallyExclusive)
  • A course may have any number of prerequisites or none? ✓ True (multiplicity *).
  • A course may not be taken with any number of courses? ✗ False (it CAN be).
  • A course may have an association relation with another course? ✓ True — reflexive association.
Ex 9 Book / Volume / Chapter (Aggregation vs Composition)

الـ scenario: Book has volumes, volumes have chapters. Chapters can be sold alone, even if book not available.

  • Fig 1 (Composition 1:* / 1:*) — ❌ wrong because composition means parts can't exist alone.
  • Fig 2 (Composition 1..* / 1..*) — ❌ wrong for same reason.
  • Fig 3 (Aggregation 1..* / 1..*) — ✅ correct — parts can exist independently.
الإجابة: Figure 3 — Aggregation is correct because chapters/volumes can be sold/exist independent of the book.
Ex 10 Event/Schedule/Display system — multiple relations

قراءة الـ diagram بدقة:

  • CurrentEventDisplay can display both SpecialEvents and RegularEvents? ✓ True — both inherit from Event, and Event is displayed.
  • CurrentEventDisplay can display multiple events? ✓ True if multiplicity supports it.
  • Schedule can contain an Announcement? ✗ False — Schedule aggregates Event, not Announcement.
  • UserInterface can display either events or announcements but not both? ✗ False — different subclasses of UI handle different types.
  • SpecialEvent should have at least one Date? Depends on multiplicity on Date association.
  • An Announcement can have a Date? ✗ False — Announcement doesn't connect to Date in the diagram.
Ex 11 Writer ↔ Document (Object diagram match)

Class diagram: Writer [*] ━ [1] Document

الـ multiplicity على Writer side هي * (يعني document واحد ممكن يكون لـ writers كتير، ومش لازم).

  • Figure I: 1 Writer ↔ 1 Document ✓ valid
  • Figure II: 2 Writers ↔ 1 Document ✓ valid (matches * to 1)
  • Figure III: 1 Writer ↔ 1 Document ✓ valid
الإجابة: Figure II is the best match for showing * multiplicity. الـ الكل valid but II demonstrates the multiplicity properly.
Ex 12 Person and Body parts (Head, Hand, Leg)
  • Fig 1 (Aggregation ◇) — ❌ wrong, body parts can't survive without person.
  • Fig 2 (Composition ◆) — ✅ correct — body parts die with person.
  • Fig 3 (Generalization) — ❌ very wrong, Head IS-NOT-A Person.
Ex 13 Car & Engine, Wheel (parts can be sold separately)

"Knowing that parts can be sold separately" → Aggregation.

  • Fig 1 (Aggregation ◇) — ✅ correct.
  • Fig 2 (Composition ◆) — ❌ wrong, parts die with whole.
  • Fig 3 (Generalization) — ❌ very wrong, Engine IS-NOT-A Car.

Exam Question Bank — Lecture 5

4 أسئلة حقيقية من الامتحانات الـ MCQ على Class Diagrams. Question 2 من كل امتحان (UML Modeling, 20 marks) بياخد كل الـ concepts دي.

Question 1 Fall 2021–22 · Q1.07 (also Winter 2021)
Given the class diagram: Advertiser [1] ━━ [*] Account. Which code properly implements it?

Code (a): Advertiser has Set accounts, addAccount() adds & sets owner with null check. Account has setOwner() that calls addAccount if newOwner != null.
Code (b): Advertiser has single Account account field. (Wrong — should be Set.)
Code (c): Advertiser has Set; Account has Set owners (multiple owners!). (Wrong — should be single owner.)
A Code (a)
B Code (b)
C Code (c)
✅ الإجابة الصح: A — Code (a) الـ multiplicity 1 to * معناها: Advertiser واحد ↔ Accounts كتير. الكود الصح:
  • Advertiser فيه Set<Account> accounts (collection).
  • Account فيه Advertiser owner (single reference, not Set).
  • addAccount + setOwner بيشتغلوا معاً للحفاظ على الـ bidirectional integrity.
B — استخدم Account account (single) في Advertiser بدل Set. غلط — كل Advertiser لازم يقدر يحمل multiple accounts. · C — جعل Account فيه Set من Advertisers (multiple owners). ده many-to-many، مش 1-to-*.
Question 2 Fall 2021–22 · Q1.08 (also Winter 2021)
Which class diagram represents this code?

class Order { OrderStatus status; ArrayList<OrderDetail> details; ... }
class OrderDetail { Product product; }
class Product { String code, name; float unitPrice; }
enum OrderStatus { OPEN, CLOSED; }
A (a) Order ◆[1:*] OrderDetail → Product; Order → OrderStatus
B (b) Order [1:1] OrderDetail
C (c) Order → OrderDetail → Product (without composition)
D (d) Order [*:*] OrderDetail (many-to-many)
✅ الإجابة الصح: A — Composition with 1:*
  • ArrayList<OrderDetail> = Order has many OrderDetails (multiplicity *).
  • Order owns OrderDetails completely (composition ◆).
  • OrderDetail has single Product reference (1:1 association).
  • Order has OrderStatus enum.
B — 1:1 غلط، الـ ArrayList معناها متعدد. · C — Order has OrderDetails as part of itself (composition), not just an association. · D — Many-to-many غلط، OrderDetail belongs to one Order only.
Question 3 Winter 2023 · Q1.04
Customers of a car sales shop can buy cars. Customers with a bad credit history should pay an extra down payment. Which diagram represents this?
A <<include>> from Bad Credit Customer's "Buy a car" to "Pay extra down payment"
B <<include>> on Customer side
C <<extend>> on Customer side
D <<extend>> from Bad Credit Customer's "Buy a car" to "Pay extra down payment"
✅ الإجابة الصح: D extend = conditional ("should pay if bad credit"). الـ Bad Credit Customer هو الـ actor المختص بالـ extra payment، فالـ extend بيكون على side بتاعه.
A, B — include = always. هنا الـ extra payment مش always (only for bad credit). · C — extend صح، بس على side بتاع Customer العادي غلط.
Question 4 Winter 2023 · Q1.05
Which class diagram is wrong?

(a) Class A with self-association (regular line with arrowhead).
(b) Class A with self-loop (curve from one end to another, plain).
(c) Class A with self-loop showing generalization arrow (open triangle).
(d) Class A with self-loop showing aggregation diamond.
A (a)
B (b)
C (c)
D (d)
✅ الإجابة الصح: C — Self-generalization is invalid Class ما تقدرش ترث من نفسها — يعني A extends A = infinite loop semantically. هذا مستحيل في OO design.
A, B — Reflexive associations valid (e.g., Course "prerequisite" Course). · D — Reflexive aggregation valid (e.g., Folder contains subfolders).
Question 5 Slide Ex 9 (practice)
A book has volumes, volumes have chapters. Chapters can be sold alone even without the book. Which is the correct relation?
A Composition (filled diamond)
B Aggregation (open diamond)
C Generalization (open triangle)
D Plain association
✅ الإجابة الصح: B — Aggregation "Parts can be sold separately" → parts can exist independently → Aggregation (open diamond ◇). لو Composition، الـ chapter ما تقدرش توجد بدون الـ book.
A — Composition means parts can't exist alone, contradicts the requirement. · C — Chapter IS-NOT-A Book. · D — Plain association لا يعبر عن "part-whole".
Question 6 Slide Ex 12 (practice)
What's the right relation between Person and body parts (Head, Hand, Leg)?
A Composition (filled diamond)
B Aggregation (open diamond)
C Generalization
D Plain association
✅ الإجابة الصح: A — Composition الـ body parts ما تقدرش تعيش بدون الـ Person. لما الـ Person يموت، الـ body parts ما تكونش entities مستقلة.
B — Aggregation معناها parts independent — مش الحالة. · C — Hand IS-NOT-A Person. · D — لا يعكس الـ "is part of".
Question 7 Practice on multiplicity
Class diagram: Project [0..*] ━━ [1..*] Employee. Which is true?
A Every project must have at least one employee, every employee must be on at least one project
B Every employee must be on at least one project; a project can exist with zero employees
C Every project must have at least one employee; an employee can exist with zero projects
D Both can exist independently
✅ الإجابة الصح: C الـ multiplicity بتقرأ على الـ end الـ opposite:
  • 1..* على Employee side → كل project لازم يكون عنده 1 employee أو أكتر.
  • 0..* على Project side → كل employee ممكن يكون مع 0 أو أكتر من المشاريع.
Question 8 Practice (Winter 2023 style)
Which of the following is NOT a valid UML relationship for a class to have with itself?
A Self-association (e.g. Person knows Person)
B Self-generalization (A extends A)
C Self-aggregation (Folder contains Folders)
D Self-composition (Tree contains Tree)
✅ الإجابة الصح: B — Self-generalization Class ما تقدرش ترث من نفسها — معناها loop infinite في الـ inheritance. الباقي valid:
  • Self-association: Course → Course (prerequisite).
  • Self-aggregation: Folder → SubFolder.
  • Self-composition: Tree → SubTree.

🚨 Common Mistakes

1. خلط Open vs Filled diamondOpen ◇ = Aggregation (independent parts). Filled ◆ = Composition (part dies with whole). تذكر: "Filled = Final".
2. قراءة Multiplicity من الجانب الغلطالـ multiplicity على side A بتقول كم instance من A لكل instance من B (الـ opposite). لو A [1] ━ [*] B: لكل B واحد، فيه A واحد بس.
3. استخدام Generalization لما Composition هي الصحHand IS-NOT-A Person. Hand IS PART OF Person. لما تشوف "part of" → فكر في Aggregation/Composition، مش Generalization.
4. Include vs Extend في Use Cases(جا في Lec 3 بس بيتسأل كتير): Include = always uses. Extend = sometimes (conditional).
5. ينساش Multiplicity على الـ endsأي association لازم لها multiplicity على الـ end (1, *, 0..1, إلخ). لو نسيت، بتفقد marks في Question 2.
