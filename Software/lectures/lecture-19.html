01

SWEBOK Software Economics

الـ software مش مجرد كود — هو investment. لازم القيمة اللي بيرجّعها تكون أكبر من تكلفته.

📍 إحنا فين دلوقتي؟ Software Economics هو آخر support pillar جنب Process Management · Project Management · Configuration Management · Quality Assurance · Professional Ethics. بيشتغل بالطول على كل مراحل المشروع (Requirements → Test).
⚖️ 1. Fundamentals of Value & Cost
  • Value vs. Cost — الـ software عبارة عن investment؛ الـ utility (الفايدة) لازم تكون أكبر من الـ effort (المجهود/التكلفة). لو مش هيرجّع قيمة أكبر من تكلفته، مش يستاهل.
  • Opportunity Cost — القيمة اللي بتخسرها لما تختار مشروع وتسيب التاني. لو شغّلت الفريق على مشروع A، فـ اللي كان ممكن تكسبه من مشروع B ده هو الـ opportunity cost.
  • Risk & Uncertainty — إدارة التأثير المالي للـ technical unknowns (الحاجات اللي لسه مش متأكدين منها تقنياً).
🔄 2. Life Cycle Economics
  • Total Cost of Ownership (TCO) — الميزانية على طول عمر المنتج، مش بس وقت التطوير: Development + Maintenance + Retirement. الـ maintenance غالباً بتكون أغلى من الـ development نفسه.
  • Sunk Costs — التكاليف اللي اتدفعت ومش هترجع. القاعدة الاقتصادية: تجاهلها وانت بتاخد قرارات المستقبل. ما تكمّلش مشروع فاشل عشان "صرفت عليه كتير" — ده تفكير غلط.
📊 3. Economic Analysis Methods
الطريقةبتقيس إيه؟
ROI (Return on Investment)نسبة الربح للتكلفة (profit-to-cost ratio).
NPV (Net Present Value)حساب الأرباح المستقبلية بقيمة النهاردة (عشان فلوس بكره مش بتساوي فلوس النهاردة).
Payback Periodالمدة لحد ما المشروع يـ "break even" (يغطّي تكلفته ويبدأ يربح).
🧭 4. Practical Strategic Decisions
  • Build vs. Buy vs. Reuse — تكتب الكود بنفسك؟ تشتري حل جاهز؟ ولا تستخدم open-source؟
  • Outsourcing — تقييم الـ cost-benefit لتشغيل عمالة خارجية.
  • Legacy Systems — قرار: نـ maintain (نصلّح) ولا wrap (نلفّه بـ interface جديد) ولا replace (نعيد كتابته من الأول)؟
💡 الخلاصة الـ economics بتجاوب على سؤال واحد: هل الفلوس والوقت اللي هندفعهم على المشروع ده هيرجّعوا قيمة أكبر؟ كل الأدوات (ROI / NPV / Payback / opportunity cost) موجودة عشان دا.

Intellectual Property & Software Protection

الـ IP laws بتحمي الإبداع (software, algorithms, designs) من الاستخدام غير المصرّح. بتشجّع الابتكار — بس ممكن كمان تقيّده.

🛡️ أنواع الـ IP الأربعة في الـ Software

© Copyright

  • بيحمي source code · UI designs · graphics · documentation من النسخ غير المصرّح.
  • بيحمي التعبير (expression) عن الفكرة — مش الفكرة نفسها.

📜 Patents

  • بتحمي الاختراعات التقنية الـ novel و الـ non-obvious + software processes.
  • شروط: Novelty + Usefulness + Non-obviousness.

™ Trademarks

  • بتحمي أسماء الـ software · اللوجوهات · brand identity.

📄 Software Licenses

  • بتحدّد ازاي ممكن الـ software يـ used · modified · shared · distributed.
⚔️ Copyright vs Patent — الفرق المهم
CopyrightPatent
بيحمي إيه؟التعبير عن الفكرة (الكود المكتوب)الـ حل التقني/process المبتكر نفسه
الشروطأصلية فقط (تلقائي بمجرد الكتابة)Novelty + Usefulness + Non-obviousness
مثالكود برنامج معيّنAmazon's One-Click Purchase patent
⚠️ التحديات الـ patents فيها legal complexity + تكلفة عالية + patent trolling (هنشوفه في Ch3).
📌 تذكرة من Lec 19 (Software Economics) قضية Oracle v. Google هي اللي blurred the line بين الكود الـ functional والـ creative. هنفصّلها في Ch4 — وهي مهمة جداً لفهم إن الـ APIs ممكن تتنسخ legally تحت fair use.
03

Software Patents — Catalyst or Cage?

الـ patents بتدفع الابتكار ولا بتعطّله؟ سيف بحدّين (double-edged sword).

📜 إيه هو الـ Software Patent؟
  • بيدّي exclusive rights للاختراع (عادةً 20 سنة).
  • بيمنع الناس التانية إنها تـ make · use · sell من غير إذن.
  • في الـ software: بيحمي كود معيّن · processes · functionality · technical methods.

✅ مع الـ Patents (Pro-Innovation)

  • أساسية لتشجيع الابتكار.
  • بتمنع المنافسين من نسخ ابتكارات كلّفت كتير.
  • بتحفّز الاستثمار في تكنولوجيا جديدة (groundbreaking).

❌ ضد الـ Patents (Stifling)

  • patents بتتدّي لأفكار obvious أو منتشرة.
  • بتؤدّي لـ "Patent Trolls" — شركات بتقاضي بدل ما تبتكر.
  • دراسة EFF: كتير من الـ patented software عام/واسع زيادة.
  • Legal minefield للمطورين الصغيرين.
📱 Case Study: Apple vs. Samsung
  • قلب القضية: Apple اتّهمت Samsung إنها خرقت patents بتاعة iPhone UI (زي الـ bounce-back effect).
  • سنة 2012: الـ jury الأمريكي وافق مبدئياً وحكم لـ Apple بـ أكتر من $1 billion.
  • النقد: features زي tap-to-zoom و rubber-banding أفكار بسيطة وبديهية — المفروض ما يتعملش ليها patent. patenting الـ basic functionalities بيضر المنافسة ويصعّب على الشركات الصغيرة الابتكار من غير خوف من القضايا.
👹 Patent Trolls
  • شركات بتشتري patents أساساً عشان تقاضي الناس أو تطلب licensing fees — مش عشان تنتج.
  • مثال: Unilock قاضت Microsoft و Google و Apple بـ patents واسعة.
  • التكتيك: استهداف الـ startups والمطورين الصغيرين اللي مش قادرين على تكاليف الدفاع القانوني.
  • الأثر: بتعطّل الابتكار وبتحوّل الموارد بعيد عن تطوير المنتج الحقيقي.
💡 الخلاصة الـ software patents = double-edged sword. لو اتساء استخدامها → بتعطّل التقدم. لو اتطبّقت بحكمة → بتغذّي الابتكار. "التحدي مش في إلغاء الـ patents، لكن في إعادة تعريفها للعصر الرقمي."

Case Study: Google v. Oracle ⭐

أشهر قضية في تاريخ الـ software copyright — استمرت 11 سنة (2010 → 2021) حوالين الـ Java APIs في Android.

📋 Google نسخت إيه بالظبط؟
  • Google ما نسختش الـ source code بتاع Java ولا الـ implementation logic.
  • نسخت الـ Structure, Sequence, and Organization (SSO) بتاع 37 Java API package.
  • يعني: class names · method names · package hierarchies — مش الكود اللي جوه.
// Java (Oracle): package java.util; public class Arrays { public static void sort(Object[] a) { ... } public static int binarySearch(Object[] a, Object key) { ... } } // Android (Google) — نفس الـ structure، implementation مختلف: package java.util; public class Arrays { public static void sort(Object[] a) { // Google's own implementation } public static int binarySearch(Object[] a, Object key) { // Google's own implementation } }
📌 ليه Google عملت كده؟ عشان تخلّي Android متاح لملايين الـ Java developers — يقدروا يستخدموا API calls مألوفة زي Arrays.sort(). الـ compatibility دي سرّعت انتشار Android.
🏢 حجة Oracle
  • الـ API structure حاجة creative expression، مش بس functional.
  • Google خرقت الـ copyright بنسخها للـ structure.
  • طلبت $9 billion تعويضات.
🟢 دفاع Google
  • الـ APIs أدوات functional — زي توزيع حروف الكيبورد أو قواعد اللغة.
  • ما ينفعش يبقى عليه copyright لو هو interface ضروري للـ interoperability.
  • الاستخدام كان transformative — عمل بيئة موبايل جديدة.
⚖️ Timeline القضية (5 جولات!)
2010
Oracle بترفع القضية
بتدّعي إن Google نسخت 37 Java API package. بتطلب $9 billion.
2012 — District Court (Judge Alsup)
Google تكسب (Round 1)
حكم إن الـ APIs ما ينفعش يتعمل ليها copyright لأنها functional.
2014 — Federal Circuit
Oracle تكسب (Round 2)
ألغى حكم District Court — قال إن الـ APIs can be copyrighted.
2016 — Jury Trial (Fair Use)
Google تكسب (Round 3)
الـ jury لقى إن استخدام Google كان fair use.
2018 — Federal Circuit (again)
Oracle تكسب (Round 4)
ألغى حكم 2016 — قال إن الاستخدام كان NOT fair use.
2020–2021 — U.S. Supreme Court
Google تكسب نهائياً (Round 5) 🏆
في 4/2021 المحكمة العليا حكمت إن استخدام Google = fair use لأنه transformative ومحدود في العناصر الـ functional.
💡 الخلاصة + Implications نسخ الـ structure عشان الـ interoperability = fair use مسموح. القضية أكّدت الـ fair use في الابتكار، وحافظت على الوصول المفتوح لـ API structures — بس سابت أسئلة مفتوحة عن حدود copyright الـ APIs في المستقبل.

Case Study: Monkey Selfie (Naruto v. Slater)

مين بيملك الـ copyright لو القرد هو اللي صوّر الصورة؟ 🐒

📸 القصة
  • مصوّر (Slater) راح إندونيسيا يصوّر قرود.
  • قرد خد الكاميرا وصوّر selfie لنفسه.
  • الراجل نشر الصورة. Wikipedia ومواقع أخدوها واستخدموها. طلب منهم يشيلوها بحجة إنه بيملكها → رفضوا.
  • PETA (منظمة حقوق الحيوان) رفعت قضية عشان القرد ياخد الـ copyright والفلوس تتصرف على القرود.
⚖️ حكم المحكمة
  • القاضي حكم إن المصوّر ما يملكش الـ copyright — لأنه مش هو اللي صوّرها بنفسه.
  • وكمان القرد ما يملكش — لأن الـ copyright للبشر مش للحيوانات.
  • النتيجة: عملياً محدش يملك الـ copyright / الصورة ممكن أي حد يستخدمها (public domain).
🎓 الدرس الـ copyright بيتطلّب مؤلف بشري (human authorship). ده سؤال مهم دلوقتي مع الـ AI-generated content برضه — مين بيملك حق الناتج؟

Software Licensing Models

الـ license بيأثّر على productivity الفريق · التكلفة · الـ scalability · والتعاون.

📄 الـ License بيحدّد إيه؟
العنصربيحدّد إيه
Ownershipمين يملكه
Accessمين يقدر يستخدمه وازاي
Deploymentفين ممكن يتنشر
Scalabilityقابلية التوسّع

الـ models الشائعة: Proprietary · Open-source · SaaS subscriptions · Freemium.

🔑 أنواع الترخيص حسب طريقة الاستخدام

Perpetual License

  • شراء مرة واحدة (one-time).
  • مفيش updates إجبارية · أدوات stable.
  • المالك بيحتفظ بكل الحقوق؛ المستخدم بيشغّل بس — ما يشوفش الكود ولا يعدّله ولا يوزّعه.
Best for: long-term projects · أمثلة: Windows, Photoshop

Subscription License

  • تكلفة متكررة (recurring).
  • updates مستمرة · scalable.
Best for: fast-paced teams

Node-Locked License

  • مربوط بـ جهاز واحد.
  • تحكّم قوي · حركة محدودة (limited mobility).
Best for: dedicated workstations

Floating License

  • pool مشترك من الـ licenses · استخدام ديناميكي.
  • كفء وقت الـ peaks.
Best for: studios & labs

Open Source Licenses ⭐

بتدّي حرية الـ use · study · modify · distribute. الـ OSI بتحتفظ بالقائمة الرسمية المعتمدة.

🪪 فئتين أساسيتين

🟢 Permissive

بتسمح باستخدام الـ software بأي طريقة تقريباً — بما فيه دمجه في proprietary software.

🔁 Copyleft

بتشترط إن الـ derivative works تتوزّع تحت نفس الـ license.

🟢 Permissive Licenses
Licenseالشرط الأساسيالاستخدام الشائع
MITتضمين الـ copyright noticeالأشهر للـ libraries والأدوات الخفيفة
Apache 2.0copyright notice + patent grantمشاريع corporate-friendly · frameworks · enterprise
BSD (2/3-clause)تضمين الـ copyright noticeBSD-origin · مشاريع أكاديمية/permissive
🔁 Copyleft Licenses
Licenseالشرط الأساسيالاستخدام الشائع
GPL v2/v3الـ derivative works لازم تكون GPL برضه (strong copyleft)Linux kernel · GNU tools
LGPLcopyleft أضعف؛ بيسمح بالـ linking من غير ما يفرض GPL على المشروع كلهlibraries واسعة الاستخدام (مثل GNU C Library)
AGPLGPL + الـ copyleft بيتفعّل بالـ network use (SaaS)web applications · SaaS platforms
🎯 اختيار License لمشروع Open Source
  • MIT / Apache 2.0 → أقصى انتشار (maximise adoption)؛ يسمح بالاستخدام في proprietary software.
  • GPL → يضمن إن كل الـ derivatives تفضل open source.
  • AGPL → يضمن إن حتى الـ SaaS deployments تنشر تعديلاتها.
🚨 مهم جداً: No license = All rights reserved لو نشرت كود من غير license، الـ copyright law بيدّي محدش الحق إنه يستخدمه — حتى لو الكود ظاهر للعامة (publicly visible). لازم تحط license صريح.
🪤 Case Study: The AGPL Trap — MongoDB & Elastic
  • MongoDB كانت بتستخدم AGPL. لكن cloud providers كانوا بيقدّموا MongoDB-as-a-service من غير ما يساهموا back → هدّد قدرتها التنافسية.
  • MongoDB حوّلت لـ SSPL (Server Side Public License) اللي بيمدّ الـ AGPL copyleft لكل الـ software المستخدم في تقديم الـ database كـ service.
  • Elastic عملت نفس الحاجة مع Elasticsearch في 2021.
  • الدرس للمهندس: لو الـ SaaS بتاعك بيعتمد على component بـ AGPL أو SSPL، الـ copyleft ممكن يجبرك تنشر كل الـ source بتاعك.
⚖️ Case Study: Enforcement — BusyBox & Android
  • الـ Software Freedom Conservancy رفعت قضايا enforcement ضد مصنّعين شحنوا Linux (GPL v2) و BusyBox (GPL v2) من غير ما يوزّعوا الـ source code المقابل — زي ما الـ GPL بتشترط. (قضايا ضد Best Buy و Samsung و شركات routers).
  • الدرس: الـ GPL compliance للـ embedded/distributed software (firmware · IoT) بيتطلّب توزيع الـ source code أو إتاحته عند الطلب الكتابي.

Software Cost Estimation ⭐

تقدير: المشروع هياخد قد إيه وقت؟ كام effort؟ كام فلوس؟ "I've got bad news and bad news!"

😅 الحقيقة المؤلمة
  • التقدير الدقيق لتكلفة المشاريع شبه مستحيل (NEARLY IMPOSSIBLE).
  • ومع ذلك، كل project manager تقريباً بيتوقّع منك تقدير دقيق.
  • الوقت الوحيد اللي تقدر تكون فيه دقيق 100% هو بعد ما المشروع يخلص خالص.
  • الـ experience بتساعد.
💵 Cost = Effort غالباً
  • معظم تكلفة الـ software هي human effort → فالـ cost estimation هي في الحقيقة effort estimation. (بس فيه تكاليف تانية: AI · cloud · tools · APIs).
  • بديهياً، الـ effort متناسب مع complexity المشروع.
  • الـ Complexity نسبية — مش بس مرتبطة بالـ size (domain · tools · language...). مشروع صغير بمهندسين جداد غير متعوّدين = معقّد. مشروع كبير بمهندسين خبرة عملوا مشاريع شبهه = مش معقّد أوي.
📐 Cone of Uncertainty

عدم اليقين (uncertainty) بيقل كل ما معلومات أكتر تتوفّر عبر مراحل المشروع:

المرحلةالمعلوماتعدم اليقين
Requirementsأقل معلوماتالأعلى
Designمعلومات أكترأقل
Developmentأكتر وأكترأقل أكتر
Testingمعظم المعلوماتمنخفض
Deliveryمعلومات كاملةالأدنى
🧠 Estimation Technique 1: Expert Judgement
  • الدقة بتعتمد على competence · experience · objectivity · perception بتاع المقدّر. "educated guess" مبني على تحليل top-down أو bottom-up + analogies.
Estimate = ( x + 4y + z ) / 6

نسأل عدة خبراء 3 تقديرات: x = متشائم (pessimistic) · y = الأرجح (most likely) · z = متفائل (optimistic). وندمجهم في beta probability distribution بالمعادلة دي.

🔮 Delphi variation الخبراء بيدّوا تقديرات سرّية ومنفصلة → بيتحسب الـ average ويتعرض على المجموعة → كل خبير يقدر يعدّل تقديره → بنكرّر لحد ما محدش عايز يعدّل. (أحياناً بيُسمح بنقاش، أو الخبراء المتطرّفين يدافعوا عن قرارهم anonymously).
🧮 Estimation Technique 2: COCOMO II (Algorithmic)

COCOMO II — estimator للـ development effort · project cost · required staff · development time. بيجاوب: "قد إيه وقت وفلوس وeffort المشروع ده هيحتاج؟". من تصميم Barry Boehm.

PM = A × (Size)E × ∏ EMi
  • PM: Person-Months (الـ effort) · A: ثابت (≈ 2.94)
  • Size: حجم المشروع بالـ KSLOC أو function points · E: scale factor (economies of scale)
  • EMi: Effort multipliers (cost drivers زي reliability · team experience · tools)
// مثال السلايد: Size = 10 KSLOC, A = 2.94, E = 1.1, ∏EM = 1.2 PM = 2.94 × (10)^1.1 × 1.2 = 2.94 × 12.59 × 1.2 ≈ 44 person-months
⚠️ ملاحظة دقّة السلايد بتكتب الناتج ≈ 41، لكن بالأرقام دي بالظبط الحساب الصح = ~44 PM (لو ∏EM كانت 1.1 بدل 1.2 يطلع ≈41 — يعني فيه rounding/typo في السلايد). المهم تفهم الـ formula مش الرقم نفسه.
🚨 نقد حديث مهم لـ SLOC-based estimation الباحثين بينتقدوا التقدير المبني على الـ SLOC (عدد السطور) لأن: الـ frameworks بتولّد كود تلقائي · الـ AI tools بتولّد كود · فيه low-code/no-code tools · اللغات بتختلف كتير في الـ verbosity (Python بيحتاج سطور أقل بكتير من Java/C++ لنفس الـ functionality).
💡 ابدأ من اليوم الأول اعمل history لمجهود التطوير في شركتك · احتفظ بـ records للـ unexpected cost factors (العميل بيغيّر رأيه · الـ environment بيتغيّر · functionality إضافية بتتطلب) · خطّط تبقى expert في تقدير الـ effort.

Question 1 Practice — economics
ما هي الـ Opportunity Cost في الـ software economics؟
A تكلفة الـ hardware والـ servers
B التكاليف اللي اتدفعت ومش هترجع
C القيمة اللي بتخسرها لما تختار مشروع وتسيب التاني
D تكلفة الصيانة على طول عمر المنتج
✅ الإجابة الصح: C Opportunity Cost = the value lost by choosing one project over another. (B هي تعريف الـ Sunk Cost، و D هي جزء من الـ TCO).
Question 2 Practice — IP
ما الفرق الأساسي بين الـ Copyright والـ Patent؟
A الـ Copyright بيحمي الأفكار نفسها، والـ Patent بيحمي الأسماء
B الـ Copyright بيحمي التعبير عن الفكرة (مش الفكرة)، والـ Patent بيحمي الحل التقني المبتكر
C الـ Copyright للـ hardware والـ Patent للـ software بس
D مفيش فرق، الاتنين نفس الحاجة
✅ الإجابة الصح: B Copyright protects the expression of ideas, not the ideas themselves. Patents protect novel + useful + non-obvious technical solutions. (A غلط لأن الـ copyright تحديداً مش بيحمي الأفكار).
Question 3 Practice — Google v Oracle
في قضية Google v. Oracle، Google نسخت إيه بالظبط من Java؟
A الـ source code والـ implementation logic بالكامل
B الـ Java Virtual Machine
C الـ Structure, Sequence & Organization (SSO) — أسماء الـ classes/methods و package hierarchy لـ 37 API package
D الـ Oracle database
✅ الإجابة الصح: C Google نسخت الـ SSO (class names, method names, package hierarchies) — مش الكود اللي جوه. المحكمة العليا 2021 حكمت إن ده fair use لأنه transformative ولأغراض interoperability.
Question 4 Practice — patents
شروط الـ Software Patent هي:
A Cheap, fast, open-source
B Popularity, profit, scalability
C Speed, security, simplicity
D Novelty, Usefulness, Non-obviousness
✅ الإجابة الصح: D الـ patent بيتطلّب يكون novel (جديد) + useful (مفيد) + non-obvious (مش بديهي). وبيدّي exclusive rights لمدة ~20 سنة.
Question 5 Practice — OSS licenses
أي عبارة صحيحة عن الـ Copyleft licenses (زي GPL)؟
A بتسمح بدمج الكود في proprietary software من غير قيود
B بتشترط إن الـ derivative works تتوزّع تحت نفس الـ license
C بتمنع تعديل الكود نهائياً
D بتشيل الـ copyright عن الكود
✅ الإجابة الصح: B Copyleft (GPL = strong copyleft) بيشترط إن الـ derivatives تفضل تحت نفس الـ license. (A ده وصف الـ Permissive licenses زي MIT/Apache). الـ AGPL بيوسّع ده للـ network/SaaS use.
Question 6 Practice — licensing
لو نشرت كودك على GitHub من غير أي license، إيه اللي بيحصل قانونياً؟
A أي حد يقدر يستخدمه ويعدّله بحرية لأنه publicly visible
B بيبقى تلقائياً MIT license
C All rights reserved — محدش له الحق يستخدمه قانونياً
D بيبقى public domain تلقائياً
✅ الإجابة الصح: C No license = all rights reserved. الـ copyright law بيدّي محدش الحق يستخدم الكود حتى لو ظاهر للعامة. لازم license صريح عشان تدّي حقوق الاستخدام.
Question 7 Practice — cost estimation
في الـ Expert Judgement، التقديرات الثلاثة (x = pessimistic, y = most likely, z = optimistic) بتتدمج بالمعادلة:
A ( x + 4y + z ) / 6
B ( x + y + z ) / 3
C ( x + z ) / 2
D x × y × z
✅ الإجابة الصح: A الـ beta probability distribution (PERT) بتدّي وزن أكبر (×4) للتقدير الأرجح: (x + 4y + z) / 6.
Question 8 Practice — Monkey Selfie
في قضية الـ Monkey Selfie (Naruto v. Slater)، ليه القرد ما ملكش الـ copyright؟
A لأن المصوّر سجّل الـ patent الأول
B لأن الـ copyright بيتطلّب مؤلف بشري — الحيوانات لا تملك copyright
C لأن الصورة كانت سيئة الجودة
D لأن Wikipedia ملكت الصورة
✅ الإجابة الصح: B المحكمة حكمت إن الـ copyright للبشر مش للحيوانات. وكمان المصوّر ما ملكهاش لأنه مش هو اللي صوّرها بنفسه — فالنتيجة عملياً محدش يملك الـ copyright.
