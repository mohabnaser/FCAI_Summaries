02

What is Modularity?

قلعة من الرمل vs قلعة LEGO. الأولى لو خربت كلها هتروح. الـ LEGO تقدر تستبدل قطعة وتعيد تشغّل الباقي.

📖 تعريف Modularity
David Parnas, 1972:"... a mechanism for improving the flexibility and comprehensibility of a system while allowing the shortening of its development time."
  • الـ Software المـ modular: الأجزاء ممكن تتبدل أو تتعاد استخدامها في software تاني.
  • الـ Modularity بتخلي الـ software أسهل في الفهم لأن كل جزء صغير يتدرس لوحده.
🧩 What is a Module?

أكتر من تعريف، كلهم بيوصلوا لنفس الفكرة:

  • "A program unit that is discrete and identifiable with respect to compiling, combining, and loading."
  • "A logically separable part of a program."
  • "A set of source code files under version control that can be manipulated as one."
  • "A collection of both data and the routines that act on it." (object-oriented view)

Module Complexity

3 مقاييس لجودة الـ module: Cyclomatic Complexity, Cohesion, Coupling.

🌪️ Module Complexity

الدرجة اللي بيها design أو code صعب الفهم بسبب:

  • كتر الـ components
  • كتر الـ relationships بينهم

أو الدرجة اللي بيها implementation صعب الـ understand and verify.

✨ Module Simplicity

العكس: الدرجة اللي بيها design أو code straightforward وسهل الفهم.

هدفنا: تصميم modules بسيطة.

📏 Measures of Module Complexity — 3 مقاييس
  1. McCabe Cyclomatic Complexity — تعقيد function واحدة.
  2. Coupling — اعتماد الـ modules على بعضها.
  3. Cohesion — تماسك الـ module من جوّه.
04

Cyclomatic Complexity — McCabe 1974 ⭐

عدد المسارات المستقلة في piece من الكود. كل ما زاد، كل ما الكود أصعب وأكتر عرضة للـ bugs.

🧮 3 طرق لحساب V(G)

1. Decisions + 1

عدد الـ binary decisions (if, while, case) + 1.

2. Closed Areas + 1

على الـ Control Flow Graph: عدد المناطق المغلقة (enclosed regions) + 1.

3. Edges − Nodes + 2P

V(G) = E − N + 2P
P = عدد الـ separate sub-graphs (عادة = 1).

⚠️ ملاحظة مهمة"Simple decision" = اختيار بين خيارين فقط. لو فيه switch بـ 3 cases، اعتبره مثل 2 simple decisions (تتفرع شجرة binary).
🔢 مثال: حساب V(G) من graph

على Control Flow Graph معين:

Edges9
Nodes7
P1
V(G) = 9 − 7 + 2(1) = 4
💻 مثال: حساب V(G) من كود
C++ · categorize()
std::string categorize(int value) {
  if (value > 100) {              // decision 1
    return "Very High";
  } else if (value > 75) {        // decision 2
    return "High";
  } else if (value > 50) {        // decision 3
    return "Medium High";
  } else if (value > 25) {        // decision 4
    return "Medium Low";
  } else if (value > 0) {         // decision 5
    return "Low";
  } else {
    return "Negative";
  }
}
// V(G) = 5 decisions + 1 = 6
📊 ايه يعتبر "كويس"؟
V(G)التقييمالتوصية
1–10Manageableسهل الفهم والاختبار. الأهداف المثلى.
11–20Moderateمحتاج testing أكتر، review عناية. ممكن refactoring.
21–50High Complexityصعب الفهم/الصيانة. غالباً candidate للـ refactor.
> 50Very Complex🚨 Red flag — لازم refactoring.
💡 توصيات McCabe
1. حافظ على V(G) < 10 خصوصاً للكود الحرج.
2. استخدم complexity كـ guideline للـ refactoring.
3. بعض الـ algorithms (statemachine, business rules) ليها complexity عالية بطبيعتها.
🛣️ العلاقة بـ Basis Path Testing

عدد test cases المطلوبة لـ full path coverage = V(G).

لو V(G) = 4، محتاج 4 test cases لتغطية كل المسارات المستقلة.

عدد test cases للـ statement coverage ممكن يكون أقل بكتير — أحياناً test case واحد بيمر بكل الـ statements (لو الكود sequential).

05

Cohesion — تماسك الـ Module ⭐

مدى ارتباط الـ functions داخل module واحد ببعضها. High cohesion = جيد. 7 مستويات من الأسوأ (Coincidental) للأفضل (Functional).

📖 ايه يعني Cohesion؟

Cohesion = درجة ارتباط الـ elements داخل module واحد ببعض. لو الـ functions كلها بتعمل حاجة واحدة related → High cohesion.

الهدف: High cohesion + Low coupling.

📈 Levels of Cohesion (من الأفضل للأسوأ)
7
🏆 Functional BEST
كل الـ elements في الـ module بتساهم في وظيفة واحدة well-defined. مثلاً: StringUtils فيه toUpper, toLower, trim — كلها string operations.
6
Sequential
الـ output من function = input للـ next. مثال: parseInput() → validate() → process().
5
Communicational
الـ functions بتشتغل على نفس الـ data. مثال: كل عمليات الـ student record (read, update, delete) في module واحد.
4
Procedural
الـ functions بتنفّذ بترتيب معين كجزء من procedure. مثال: open file → read → close.
3
Temporal
الـ functions بتنفّذ في نفس الوقت بس مش related. مثال: initializeApp() اللي بيعمل load للـ configs, connect للـ DB, start للـ logger كلها في الـ startup.
2
Logical
الـ functions بتعمل حاجات similar logically بس بـ implementations مختلفة. مثال: handleInput(type) اللي بيعمل switch على type والـ handlers مختلفة جداً.
1
😈 Coincidental WORST
الـ elements مالهاش علاقة ببعض. اتحطوا في module بالصدفة. مثال: ملف فيه parseDate() و sendEmail() و encryptPassword().
📌 Winter 2025 Q1.06لو عندك StringUtils فيها static functions كلها بتعالج Strings — الجواب: Functional cohesion (الأفضل).
🧠 Mnemonic

من الأفضل للأسوأ:

Functional > Sequential > Communicational >
Procedural > Temporal > Logical > Coincidental

أو: Fancy Sandwich Cooks Prefer Tasty Lemon Cake 🍰

06

Coupling — اعتماد الـ Modules على بعضها ⭐

مدى اعتماد module على module تاني. Low coupling = جيد. الدكتور بيذكر 5 أنواع: Common Environment · Content · Control · Data · Pathological.

📖 ايه يعني Coupling؟

Coupling = درجة الاعتماد بين modules مختلفة. لو module بيعرف تفاصيل داخلية في module تاني → High coupling (مش جيد).

الـ goal: كل module مستقل بقدر الإمكان — لو غيرت module واحد، الباقي ما يتأثرش.

📉 5 أنواع Coupling (حسب السلايدات)
النوعالتصنيفالوصف
😈 ContentHigh — WorstModule بيـ access/modify مباشرةً للـ internals بتاع module تاني. أعلى درجة coupling. بيكسر الـ encapsulation. fragile جداً.
PathologicalHighModule بيعتمد على الـ internal implementation بتاع module تاني / متشابكين بشدة. سلوك unpredictable. مؤشر على تصميم سيئ.
Common EnvironmentMediumالـ Modules بتشترك في global data / environment. أي تعديل في الـ global بيأثر على الكل. بيقلّل الـ encapsulation.
ControlMediumModule بيتحكم في module تاني عن طريق flags / control variables. مرونة في التنفيذ بس بتقلّل الـ clarity.
🏆 DataLow — Bestالـ Modules بتتواصل عن طريق parameters بس. مفيش shared internal state. النوع المرغوب.
📊 ملخص الدكتور (3 tiers) Low (Data) = جيد، بيحسّن الـ maintainability/modularity · Medium (Common Environment, Control) = tradeoffs مختلطة · High (Content, Pathological) = fragile وصعب الصيانة.
💻 مثال على Common Environment Coupling
JavaScript — Bad coupling
let counter = 0;             // ⚠️ Global shared state

class ModuleA {
  increment() { counter++; }   // writes global
}

class ModuleB {
  display() {
    console.log("Count: " + counter);  // reads global
  }
}
📌 Winter 2025 Q1.07الكود ده Common Environment Coupling — الـ modules بتشترك في counter global. لو ModuleC ضيفته بتمسح الـ counter، ModuleA و ModuleB كلهم هيتأثروا.

✅ الحل: Data Coupling

JavaScript — Good coupling
class Counter {
  constructor() { this.value = 0; }
  increment() { this.value++; }
  get() { return this.value; }
}

function display(count) {     // only receives what it needs
  console.log("Count: " + count);
}
☠️ مثال على Pathological Coupling

module خارجي بيتحكم في الـ internal state بتاع module تاني، والسلوك بيعتمد على الـ state ده → unpredictable.

C++ — Pathological
class ModuleA {
  int state = 0;
public:
  void setState(int s) { state = s; }
  void execute() {
    if (state == 1)      { /* action 1 */ }
    else if (state == 2) { /* action 2 */ }
    else               { /* default */ }
  }
};

class ModuleB {
public:
  void changeModuleAState(ModuleA& a, int n) {
    a.setState(n);   // ⚠️ B يتحكم في state بتاع A من بره
  }
};

// main: B بيغيّر حالة A قبل ما A.execute() — سلوك A بيعتمد على B
moduleB.changeModuleAState(moduleA, 1);
moduleA.execute();
🔴 ليه ده Pathological؟سلوك ModuleA.execute() بيعتمد كلياً على state اتحطّ من module تاني (ModuleB). الاتنين متشابكين، وأي تغيير في تسلسل الـ calls بيغيّر النتيجة بشكل غير متوقع.
🎯 The Golden Rule
High Cohesion + Low Coupling

كل module:

  • ✅ عنده مسؤولية واحدة واضحة (High cohesion - Functional).
  • ✅ بيعتمد على باقي الـ modules بأقل قدر (Low coupling - Data only).
  • ✅ سهل تستبدله، تختبره، تفهمه لوحده.

Exam Question Bank — Lecture 4

4 أسئلة حقيقية من Winter 2025 على Cyclomatic Complexity, Cohesion, Coupling, White-box Testing.

Question 1 Winter 2025 · Q1.05
What is the average Cyclomatic Complexity per function in class StringUtils?
(The class has multiple static String methods, each with at most 1 if/branch.)
A 2
B 3
C 4
D 6
E 12
✅ الإجابة الصح: A — 2 كل function فيها branch واحد بسيط (1 decision)، فالـ V(G) لكل function = 1 + 1 = 2. الـ average هو 2.
الإجابات التانية بتفترض branches أكتر مما الكود حقيقي بيحتوي. لو الـ function ليها n simple decisions، V(G) = n+1.
Question 2 Winter 2025 · Q1.06
What type of cohesion does StringUtils have? (StringUtils contains multiple static methods all dealing with String operations: toUpper, toLower, etc.)
A Functional
B Sequential
C Temporal
D Communicational
✅ الإجابة الصح: A — Functional كل الـ functions في StringUtils بتساهم في وظيفة واحدة well-defined: String manipulation. ده الـ best level of cohesion.
B (Sequential) — لما output method بيكون input للتاني. هنا الـ functions independent. · C (Temporal) — لما الـ functions بتنفذ في نفس الوقت (مثل init). · D (Communicational) — لما الـ functions بتشتغل على نفس الـ data. هنا كل function بتاخد String مختلف.
Question 3 Winter 2025 · Q1.07
What kind of coupling is shown in this JS code?

let counter = 0;
class ModuleA { increment() { counter++; } }
class ModuleB { display() { console.log("Count: " + counter); } }
A Data
B Content
C Common (Environment)
D Control
✅ الإجابة الصح: C — Common Environment الـ counter هو global variable بتشارك فيه ModuleA و ModuleB. أي module يعدّل في الـ counter يتأثر الباقي. ده exactly تعريف الـ Common Coupling.
A (Data) — لو الـ counter كان parameter بيتمرر للـ function، كان Data coupling. · B (Content) — لو module كان بيدخل على internals لـ object/class تاني مباشرةً (أسوأ نوع). · D (Control) — لما function بتمرر flag للتانية تتحكم في الـ logic.
Question 4 Winter 2025 · Q1.03
What is the minimum number of test cases needed to achieve:
(1) FULL statement coverage, AND
(2) FULL path coverage (not branch coverage)
for a sequential code block with one chained if-else-if structure with 4 distinct paths?
A 2 for statement, 4 for path
B 1 for statement, 2 for path
C 2 for statement, 2 for path
D 1 for statement, 4 for path
✅ الإجابة الصح: D — 1 for statement coverage, 4 for path coverage
  • Statement coverage: لو الكود فيه عبارات executed مرة على الأقل بـ input واحد، test case واحد كفاية.
  • Path coverage: محتاج test case لكل independent path. عدد الـ paths = V(G). لو V(G) = 4، محتاج 4 test cases.
الـ statement coverage دايماً ≤ path coverage. الإجابات اللي بتسوي بينهم أو تقلب الترتيب غلط.

🚨 Common Mistakes

1. خلط ترتيب CohesionFunctional هي الـ best — مش Coincidental. الـ ladder صعود من coincidental لـ functional.
2. خلط ترتيب CouplingData هو الـ best — لإن المعلومات اللي بتمرر بس اللي محتاجها. Content هو الأسوأ (access internals).
3. نسيان Pathological couplingالدكتور بيذكر 5 أنواع: Common Environment · Content · Control · Data · Pathological. (مش External/Stamp — دول من taxonomies تانية مش في سلايداته).
4. V(G) = decisions، مش decisions + 1الصيغة هي decisions + 1. مش بس عدد الـ ifs.
5. تخيل إن switch بـ 3 cases = 1 decisionلأ — كل case = decision لوحده تقريباً (binary tree of decisions). switch بـ N cases ≈ N-1 simple decisions.
