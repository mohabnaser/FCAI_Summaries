01

Why Model Software?

Modeling يعني بناء abstraction للواقع: بنتجاهل التفاصيل غير المهمة ونحتفظ بالمهمة.

📖 What is Modeling?
  • Modeling = building an abstraction of reality.
  • Abstractions = simplifications because:
    • بنتجاهل irrelevant details.
    • بنركز على relevant details بس.
  • معنى Relevant بيتحدد حسب هدف الـ model.
📈 Why Model Software?
  • الـ Software بقى أكتر تعقيداً — Windows 10 أكتر من 50 مليون سطر كود.
  • مبرمج واحد ما يقدرش يفهم كل الكود ده لوحده.
  • الكود مش سهل الفهم للمطورين اللي ما كتبوش.
  • نحتاج تمثيلات بسيطة لأنظمة معقدة.
🌐 Application Domain vs Solution Domain

🏢 Application Domain

الـ environment اللي النظام بيشتغل فيه — مثلاً Air Traffic Control: الطيارات، الأبراج، الرادار.

→ يستخدم في Requirements Analysis.

💻 Solution Domain

الـ technologies المتاحة لبناء النظام — Classes, Databases, APIs, frameworks.

→ يستخدم في System Design & Object Design.

03

Interaction Diagrams

Sequence + Communication diagrams. بنستخدمهم لنمذجة الـ dynamic aspects.

🎬 What are Interaction Diagrams?

بتُستخدم لنمذجة الـ dynamic aspects للنظام:

  • بتعرض إزاي النظام بيشتغل.
  • غالباً بتُبنى من use case + class diagram.
  • الهدف: نوضح إزاي مجموعة objects بتنفذ الـ interactions المطلوبة مع actor.
📦 2 Types

📈 Sequence Diagram

أكتر شيوعاً. بيركز على الترتيب الزمني للـ messages. المحور الرأسي (vertical axis) = الزمن.

🕸️ Communication Diagram

بيركز على الـ relationships بين objects. الزمن مش explicit، وبدل كده بنرقّم الـ messages.

🧱 Elements of Sequence Diagrams
  • Instances of classes — boxes فيها object identifier تحته خط (underline).
  • Actors — stick figures زي Use Case diagrams.
  • Messages — أسهم من actor لـ object، أو من object لـ object.
  • Lifelines — خطوط منقطة (dashed vertical lines) ممتدة من كل box.
  • Activation boxes — أعمدة طويلة على الـ lifeline بتبين الفترة اللي الـ object نشط فيها.

05

Message Types — 4 Types

الدكتور بيحب يسأل عن الفرق بين Synchronous, Asynchronous, Create, Reply.

🔄 Synchronous Message

──────▶ (solid arrow, filled triangle)

الـ Sender بيستنى لحد ما الـ message تتعالج قبل ما يكمل. دي Wait semantics، وده هو الـ method call العادي.

⚡ Asynchronous Message

──────▷ (solid arrow, open arrowhead)

الـ Sender ما يستناش. دي No-wait semantics. مفيش explicit return، وده بيسمح للـ objects تشتغل concurrently.

↩️ Reply Message

⇠⇠⇠⇠⇠ (dashed arrow, open arrowhead)

دي الـ return value من message سابقة. بترسم كـ dashed line. ممكن تتجاهلها لو الـ flow واضح.

🆕 Create Message

⇢⇢⇢⇢⇢ (dashed arrow → object box)

بيخلق object جديد أثناء الـ interaction. في Java = constructor call (new ClassName()). الـ arrowhead بيشاور للـ object الجديد.

💡 الفرق المهم Synchronous = wait. Asynchronous = no wait. لو شفت filled arrowhead يبقى synchronous (method call). ولو شفت open arrowhead + dashed يبقى reply/async.
06

Combined Fragments — opt, loop, alt

الـ Frames دي بتغلف أجزاء من الـ sequence للتحكم في الـ control flow.

❓ opt — Optional

الـ messages داخل الـ frame بتتنفذ لو الـ guard condition = true. زي if.

opt
[hasPrerequisite]
create new :Registration
:Registration → :Student : addToSchedule
:Registration → :CourseSection : addToRegistrationList

لو الـ student ما عندوش الـ prerequisite، الـ frame ده بيتخطى بالكامل.

🔁 loop

الـ messages بتتنفذ عدة مرات. ممكن تحط range زي 0..n أو *.

loop 0..numPurchases
:Bill → :Purchase : getSubtotal
:Purchase → :Item : getUnitPrice
🔀 alt — Alternative (if/else)

اختيار بين multiple paths بناءً على شروط. زي if/else أو switch.

alt
[credentials valid]
redirect to dashboard
[else]
show error message
📌 Other useful frames
par — Parallel (concurrent execution).
break — Like break in a loop, exits the current scope.
ref — الرجوع لـ sequence diagram تانية (reusable).
07

Object Creation & Destruction

إظهار حياة الـ object: من لحظة إنشائه لحد تدميره.

🌱 Element Creation

الـ Object الجديد بيظهر في النص أو تحت، مش فوق مع الـ objects الموجودة من البداية.

السهم dashed arrow with open arrowhead بيشاور للـ box الجديد. الـ box هو اللي السهم بيوصله.

في الكود: ده بيمثل new ClassName().

💀 Element Destruction

الـ Object بينتهي بعلامة large X symbol على الـ lifeline عند نقطة الـ destruction.

الـ message اللي بتدمر الـ object بتخلص عند علامة X.

مثال: cancelBooking → cancel → :Booking ينتهي بـ X.

08

Library System — Full Example ⭐

الدكتور عمل sequence diagram لـ Issue a Book مع opt fragment. ركز عليه.

📚 Use Case: Issue Book

Actor: Librarian · Goal: Issue a book to a borrower if his status permits.

Main Scenario:

ActorSystem
1- Enter user's ID2- Display info & user status
3- If allowed to borrow, enter book copy ID4- Create borrowing record
5- Update user's #books borrowed
6- Update book's availability

Exceptional Flow — Suspended/MaxBorrowed:

If user is suspended or at max → deny issuing.

📈 Sequence Diagram Flow

الـ Objects المشاركة (من الـ classes الموجودة):

  • Librarian — actor
  • GUI — boundary
  • LibrarySystem — control
  • Catalog — entity
  • Book / BookCopy — entity
  • User / Record — entity

Step-by-step:

  1. Librarian → GUI : checkUserStatus(ID)
  2. GUI → LibrarySystem : checkUserStatus(ID)
  3. LibrarySystem → User : checkStatus() → returns status
  4. Return "Allowed" to Librarian
  5. opt [if Allowed]:
    1. Librarian → GUI : IssueBook(title, ID)
    2. GUI → LibrarySystem : SearchCatalog(title, auth)
    3. LibrarySystem → Catalog : findBook(title, auth) → returns Book
    4. LibrarySystem → Book : getCopyInfo()
    5. LibrarySystem → User : updateBorrowed() (self message)
    6. LibrarySystem → BookCopy : updateAvailability()
💡 المفاتيح للنجاح
  1. كل step في الـ use case = message أو اتنين في الـ sequence.
  2. كل method جديدة لازم تكون في الـ class diagram.
  3. أسماء الـ objects لازم تكون underlined.
  4. استخدم opt للـ conditional flow.
🚨 أخطاء الطلبة المتكررة
  • ❌ نسيان underline لأسماء الـ objects.
  • ❌ استخدام method names مش موجودة في الـ class diagram.
  • ❌ نسيان الـ return messages (الـ dashed lines).
  • ❌ ترتيب الـ time غلط (الزمن بيمشي من فوق لتحت).
  • ❌ خلط الـ actor بـ object — Actor = stick figure، و Object = box.
  • ❌ نسيان الـ opt frame للـ conditional logic.
09

Why Not Just Code It?

سؤال طبيعي: لو الـ sequence diagram قريب من الكود، ليه نرسمه أصلاً؟ عندنا 4 أسباب.

1️⃣ Above the level of real code

الـ sequence diagram بيركز على الـ interactions، مش تفاصيل الـ syntax. اللي عايز يشوف "إزاي الـ objects بتتكلم" هيفهمه بسرعة.

2️⃣ Language-independent

نفس الـ sequence diagram ممكن يتنفذ في Java, Python, C#, etc.، والـ design بيفضل محفوظ بغض النظر عن الـ implementation.

3️⃣ Non-coders can read/do them

Business analysts, project managers, clients يقدروا يفهموا الـ flow ويراجعوا الـ design من غير ما يكونوا programmers.

4️⃣ Visual representations

الـ Visual representation أسرع في الفهم من النص. screenshot واحد ممكن يختصر ألف سطر كود.

⚠️ الـ Risks الـ Dynamic modeling صعب. في systems كبيرة فيه كم هائل من الـ paths. الحل يكون iteratively: ابدأ بـ class diagrams + use cases + interaction diagrams + state diagrams، وراجعهم كلهم مع بعض.

Practice Quiz — Lecture 6

8 أسئلة على Sequence Diagrams. الـ Question 2 في كل امتحان final فيه 4 marks على sequence diagram. ركز.

Question 1 Practice — concept
What does the vertical dimension represent in a sequence diagram?
A Sequence of classes
B Time
C Number of messages
D Object visibility
✅ الإجابة الصح: B — Time الـ vertical (top to bottom) = الزمن. والـ horizontal = الـ objects المختلفة.
A — الـ classes بتظهر أفقياً مش رأسياً. · C, D — مش معنى الـ vertical dimension.
Question 2 Practice — message types
A synchronous message means:
A The sender does not wait for any response
B The message creates a new object
C The sender waits for the message to be handled before continuing
D The message returns a value
✅ الإجابة الصح: C — wait semantics Synchronous = wait. ده الـ method call العادي.
A — ده Asynchronous. · B — ده Create. · D — ده Reply.
Question 3 Practice — frames
A team is modeling a login flow where credentials are checked, and if valid, user is sent to dashboard, otherwise error is shown. Which frame should be used?
A alt (alternative)
B loop
C par (parallel)
D ref (reference)
✅ الإجابة الصح: A — alt Two distinct paths (valid/invalid) = alt (if/else).
B (loop) — تكرار. هنا مش تكرار. · C (par) — concurrent execution. · D (ref) — reference to another diagram.
Question 4 Practice — lifeline names
In a sequence diagram, you see : Lecturer. What does this mean?
A Object named "Lecturer"
B Anonymous object of class Lecturer
C Object Lecturer at index 0
D Class named Lecturer
✅ الإجابة الصح: B — Anonymous instance of class Lecturer Empty name + :Class = anonymous instance.
A — كان لازم يكون Lecturer بدون نقطتين. · C — الـ index بيتكتب lecturer[0]. · D — في sequence diagram لازم تكون instances مش classes.
Question 5 Practice — destruction
How is the destruction of an object shown on its lifeline?
A Lifeline ends with a square box
B Lifeline becomes a dashed red line
C The object name is crossed out
D A large X symbol at the end of the lifeline
✅ الإجابة الصح: D — Large X at end of lifeline مثال: في cancel booking، الـ :Booking object lifeline بينتهي بـ X.
Question 6 Practice — modeling concepts
Which is NOT a good reason to draw a sequence diagram rather than just writing code?
A Sequence diagrams are language-independent
B Non-coders can read them
C They are faster to execute than real code
D They provide a visual representation
✅ الإجابة الصح: C — sequence diagrams DON'T execute Sequence diagrams هي design artifacts، مش executable code. أهدافها: مستقلة عن اللغة، يفهمها non-coders، تمثيل بصري، ومستوى أعلى من الكود.
Question 7 Practice — create
Which symbol represents object creation in a sequence diagram?
A Solid arrow with filled triangle
B Dashed arrow with open arrowhead pointing to the new object box
C Large X at start of lifeline
D Double-headed arrow
✅ الإجابة الصح: B Create = dashed arrow with open arrowhead بيشاور على box الـ object الجديد. مماثل لـ new ClassName() في Java.
A — ده synchronous message عادي. · C — علامة X بتشير لـ destruction مش creation.
Question 8 Practice — sequence + class
Which of these MUST you have before drawing a sequence diagram?
A A complete state diagram
B The full source code
C A deployment diagram
D A use case and a class diagram
✅ الإجابة الصح: D — Use case + Class diagram الـ Sequence diagram بينفّذ use case معين، وبيستخدم الـ methods من الـ class diagram. لازم الاتنين يكونوا موجودين قبله.
A — الـ State diagram اختياري ومفيد لـ objects عندها complex states. · B — الـ sequence diagram بيسبق الكود غالباً، مش العكس. · C — الـ Deployment diagrams تخص الـ infrastructure، مش requirement.

🚨 Common Mistakes

1. Methods مش موجودة في الـ class diagramكل method في الـ sequence diagram لازم تكون موجودة في الـ class diagram. لو ضفت method جديدة، ارجع وضيفها في الـ class.
2. نسيان underline لأسماء الـ objectsالـ Object box في sequence diagram اسمه دايماً underlined لأنه instance مش class.
3. خلط arrow typesSolid filled = sync · Solid open = async · Dashed = reply · Dashed to box = create.
4. ترتيب الزمن غلطالزمن بيمشي من فوق لتحت. لو رسمت msg أعلى من msg تاني، الأعلى بيحصل أولاً.
5. نسيان opt/alt للـ conditional flowلو في if في الـ use case، لازم تكون opt أو alt. ما تتجاهلهاش.
