02

Software Process Models

كل process model بيوصف: activities · order · roles · deliverables.

📖 Definition

A software process model describes:

  • Activities performed to produce software.
  • The order of these activities.
  • The roles responsible for each.
  • The deliverables (artifacts) from each.
🌟 الـ 5 Main Models

🌊 Waterfall

Linear · phase-by-phase

🔂 Iterative / Incremental

Mini-waterfalls per increment

🚀 Agile

Continuous · flexible

🏗️ RUP

Iterative framework

⚡ Scrum

Sprints · roles · ceremonies

📌 Winter 2023 Q1.06"What is true about software process models?"
الإجابة الصح: All three — (I) Include linear and agile · (II) Define who/what/when/artifacts · (III) Different ways to organize SW dev activities.
03

Waterfall Model

The classic linear model. Easy to understand, strict to enforce.

🌊 The Stages
5. Testing
4. Implementation
3. Design
2. Analysis
1. Requirements

Each phase produces a frozen deliverable before the next begins. No going back.

✅ Strengths
  • Easy to understand, easy to use.
  • Provides structure to inexperienced staff.
  • Milestones well understood.
  • Sets requirements stability.
  • Good for management control (plan, staff, track).
  • Works well when quality > cost/schedule.
❌ Deficiencies
  • All requirements must be known upfront.
  • Deliverables are frozen — no flexibility.
  • Can give false impression of progress.
  • Doesn't match real problem-solving nature.
  • Integration is one big bang at the end.
  • Customer can't preview the system early.
🎯 When to Use Waterfall
  • Requirements are very well known.
  • Product definition is stable.
  • Technology is understood.
  • New version of an existing product.
  • Porting an existing product to a new platform.
04

Incremental / Iterative Model

Multiple mini-waterfalls. Each increment delivers functioning software.

🔂 The Idea
  • Construct a partial implementation of the system.
  • Slowly add more functionality over time.
  • Prioritize requirements and implement them in groups.
  • Each release adds functions to previous, until complete.
✅ Strengths
  • Develop high-risk or major functions first.
  • Each release = operational product.
  • Customer can respond to each build.
  • Lowers initial delivery cost.
  • Initial product delivery is faster.
  • Customers get important functionality early.
  • Risk of changing requirements is reduced.
❌ Weaknesses
  • Requires good planning & design.
  • Need complete & fully functional system early-defined.
  • Well-defined module interfaces required.
  • Total cost of complete system not lower.
05

Agile SDLCs

Iterative · Incremental · Adaptive planning · Flexible to change.

🚀 Agile Properties
  • Usually less formal + reduced scope.
  • Reduce documentation.
  • Embed quality activities in the development process.
  • Used for time-critical applications.
📜 12 Agile Principles (Agile Manifesto)
  1. Highest priority: satisfy customer through early & continuous delivery.
  2. Welcome changing requirements, even late in development.
  3. Deliver working software frequently (weeks vs months — prefer shorter).
  4. Business people + developers work together daily.
  5. Build around motivated individuals — give them environment + support + trust.
  6. Face-to-face conversation = most efficient method.
  7. Working software = primary measure of progress.
  8. Promote sustainable development pace.
  9. Continuous attention to technical excellence + good design.
  10. Simplicity — maximizing work NOT done.
  11. Best architectures/requirements/designs emerge from self-organizing teams.
  12. Team reflects regularly on how to improve.
🎭 Agile Methods
  • ASD — Adaptive Software Development
  • FDD — Feature Driven Development
  • Crystal Clear
  • DSDM — Dynamic Software Development Method
  • RAD — Rapid Application Development
  • Scrum ⭐
  • XP — Extreme Programming
06

RUP — Rational Unified Process

Iterative framework by IBM. Adaptable, not prescriptive.

📊 RUP — 4 Phases
  1. Inception — Scope, vision, business case.
  2. Elaboration — Architecture, requirements refinement.
  3. Construction — Build the system.
  4. Transition — Deliver, train, deploy.

Within each phase, multiple iterations. Each iteration involves Business Modeling, Requirements, Analysis, Design, Implementation, Test, Deployment — but with varying intensity.

👥 RUP Roles (UPEDU)

Analyst · Designer · Implementer · Integrator · Tester · Change Control Manager · Configuration Manager · Project Manager · Reviewer · Stakeholder

07

Scrum ⭐⭐⭐

الـ exam بيركز جداً عليه. 4-5 أسئلة في كل امتحان عن الـ Scrum.

⚡ Scrum Elements
  • Gathering Requirements — User Stories, Product Backlog.
  • Team Roles — 3 roles.
  • Release Planning — Pick stories, prioritize, estimate.
  • Sprints — 3-30 days, usually 2 weeks.
  • Daily Scrum Meeting — 15 min standup.
  • Sprint Retrospective — End of sprint reflection.
  • Burndown Chart — Track progress.
👥 3 Scrum Roles
🦸

Scrum Master

The team leader. Facilitates the process, removes blockers, protects the team. NOT a manager.

👔

Product Owner

Represents the customer/business. Owns the product backlog, prioritizes stories.

👨‍💻

Team (Everyone Else)

Cross-functional members: devs, designers, testers. Self-organizing. NO sub-roles inside.

📌 Fall 2021-22 Q1.09"Team leader in Scrum" = Scrum Master. (Not project manager, not client rep.)
📌 Winter 2021 Q1.09"Roles in Scrum team" = Scrum master, product owner, and everyone else.
🏃 Sprints

Split the release into sprints (typically 2 weeks). Each sprint delivers ship-ready user stories.

☀️ Daily Standup Meeting
  • Time: 15 minutes max.
  • When: Every morning before work.
  • Standing up (to keep it short).

3 questions for each team member:

  1. What did you do yesterday?
  2. What will you do today?
  3. What obstacles do you face?
📌 Winter 2021 Q1.10الإجابة: (1) What did you do yesterday? (2) What will you do today? (3) What obstacles do you face?
🔄 Sprint Retrospective

Happens at the END of each sprint. The team reflects:

  • What was good?
  • What was bad?
  • How can we improve?
📌 Fall 2021-22 Q1.10Retrospective is done at the end of the sprint to reflect on what went well and what can be improved.
📌 Winter 2025 Q1.13What is NOT true about Scrum Retrospective? — It discusses the plan of the next release with the client. The retrospective is internal to the team, not with client.
📡 How Scrum Achieves Quick Client Communication

From Winter 2023 Q1.07:

  • ✅ Short release cycles
  • ❌ Daily stand-up (this is INTERNAL, not with client)
  • ✅ Product Owner role (represents the client in the team)

Answer: Only I and III (short release cycles + product owner role).

Question 1 Fall 2021–22 · Q1.09
The team leader in Scrum process is called:
A Product owner
B Project manager
C Scrum master
D Scrum supervisor
E Client representative
✅ الإجابة الصح: C — Scrum Master Scrum Master = facilitator/team leader. NOT a project manager (Scrum has no PM role). The Product Owner represents the client.
Question 2 Fall 2021–22 · Q1.10
Scrum retrospective meeting is done:
A Every day for 15 minutes to discuss progress and plan of the day
B At the beginning of the sprint to discuss priorities
C At the end of the sprint to reflect on what went well and what can be improved
D At the beginning of the project to discuss scope/duration/budget
✅ الإجابة الصح: C — End of sprint, reflection (A) describes the daily standup, not retrospective.
Question 3 Winter 2021 · Q1.09
The roles in the Scrum team are:
A Requirements engineer, programmer, manager
B Scrum master, product owner, and everyone else (other team members)
C Business analyst, Scrum developer, tester
D Scrum analyst, client, software engineer
✅ الإجابة الصح: B Scrum has exactly 3 roles: Scrum Master, Product Owner, Team. The team is cross-functional with NO sub-roles inside.
Question 4 Winter 2021 · Q1.10
Scrum daily meeting is done standing for 15 minutes to ask these questions for every team member:
A (1) What did you do yesterday? (2) What obstacles did you face? (3) What caused these obstacles?
B (1) What stories did you finish? (2) Which next? (3) How long?
C (1) What did you do yesterday? (2) What will you do today? (3) What obstacles do you face?
D (1) What did you do yesterday? (2) What today? (3) Who was with you?
✅ الإجابة الصح: C — Yesterday / Today / Obstacles الـ 3 questions الكلاسيكية للـ daily standup.
Question 5 Winter 2023 · Q1.06
What is true about software process models?
I. They include linear models like waterfall and agile models like XP.
II. They define who does what and when to produce which artifacts.
III. They are different ways for organizing the software development activities.
A Only I
B Only I and III
C Only II
D Only II and III
E All three
✅ الإجابة الصح: E — All three كل العبارات صحيحة. Process models = ways to organize, define who/what/when, and span from linear to agile.
Question 6 Winter 2023 · Q1.07
How does Scrum achieve effective and quick communication with the client?
I. By adopting short release cycles
II. By holding a daily stand-up meeting
III. By having a product owner role
A Only I
B Only I and II
C Only II
D Only II and III
E Only I and III
✅ الإجابة الصح: E — Only I and III Short release cycles let the client see progress quickly. Product Owner is the dedicated client liaison.
II (Daily standup) — INTERNAL meeting, client is NOT part of it.
Question 7 Winter 2025 · Q1.12
Given diagrams: (1) staircase rising, (2) iterations with feedback at each, (3) many small repeated mini-cycles — which model is which?
A (1) Waterfall, (2) Agile, (3) Iterative
B (1) Agile, (2) Iterative, (3) Waterfall
C (1) Waterfall, (2) Iterative, (3) Agile
D (1) Iterative, (2) Agile, (3) Waterfall
✅ الإجابة الصح: C Staircase = Waterfall (linear). Few iterations = Iterative. Many tiny cycles = Agile.
Question 8 Winter 2025 · Q1.13
What is NOT true about Scrum Retrospective meeting?
A It discusses the plan of the next release with the client
B It discusses what went good or bad in the last sprint
C It discusses how we can improve in the next sprint
D It is held after the sprint is finished
✅ الإجابة الصح: A Retrospective is internal to the team — not with client. The client involvement comes through other means (Sprint Review, Product Owner). B, C, D are all true.

🚨 Common Mistakes

1. Scrum Master ≠ Project ManagerScrum has NO PM role. Scrum Master facilitates, doesn't direct.
2. Daily Standup ≠ Client CommunicationIt's INTERNAL only. Client communication is via short release cycles + Product Owner.
3. Retrospective ≠ Sprint PlanningRetrospective = improving the process. Sprint Planning = picking stories. Both are separate.
