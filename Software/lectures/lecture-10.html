📜 Operation Contract — Step 1 Detail

الـ method interface = name + parameters + return + visibility. الـ functionality بتحدد عن طريق contract:

  • Precondition — قيود على attributes والـ actual parameters قبل الـ call.
  • Postcondition — effects بعد الـ call: instance creation/deletion · attribute modification · association formed/broken.
BankAccount.withdraw() contract
// Interface
public void withdraw(float amt) { ... }

// Contract
// Precondition:  amt > 0, amt <= balance
// Postcondition: new balance = old balance - amt
02

Order of Implementation

Top-Down vs Bottom-Up.

⬇️ Top-Down

ابدأ بالـ high-level components (e.g., OrderManager) → بعدين الـ dependent components.

✅ Advantages

  • Class A = higher level component، makes use of others.
  • Early validation of overall system.

❌ Disadvantages

  • محتاج stubs (temporary implementations) للـ lower classes.
⬆️ Bottom-Up

ابدأ بالـ lower-level classes (e.g., Item, Product) → بعدين الـ higher-level.

✅ Advantages

  • Low-level classes stand alone.
  • مفيش stubs محتاجة.

❌ Disadvantages

  • الـ complete executable يتأخر للنهاية.
💡 الـ practical answerغالباً hybrid: ابدأ بـ utility/value classes (Bottom-Up partially)، بعدين main flow Top-Down مع stubs بسيطة.

05

Unidirectional Associations

Associations في UML → references في Java. الـ default multiplicity = "0..1" (reference can be null).

🔗 Optional 0..1 Association

UML: Account [1] ━→ [0..1] DebitCard

public class Account {
  private DebitCard theCard;  // 0..1 = good match for ref

  public DebitCard getCard() { return theCard; }
  public void setCard(DebitCard card) { theCard = card; }
  public void removeCard() { theCard = null; }
}
🔒 Immutable Association {Immutable}

الـ link مرة واحدة set ومش بيتغير. لازم explicit check:

public void setCard(DebitCard card) {
  if (theCard != null) {
    // throw ImmutableAssociationException
  }
  theCard = card;
}
❗ Compulsory Association (multiplicity 1)

الـ link لازم مش null أبدًا. enforce في الـ constructor:

public Account(Guarantor g) {
  if (g == null) {
    // throw NullLinkError
  }
  theGuarantor = g;  // must have a guarantor
}
06

Bidirectional Associations ⭐

اللي بتيجي في Question 1.07 من امتحانات Fall 2021-22 / Winter 2021.

↔️ Implementation Strategy

References في Java unidirectional. علشان bidirectional محتاج 2 references — واحد في كل class.

الـ key concept: Referential Integrity — الـ links لازم تكون consistent.

⚠️ Referential Integrity Violation

Account acc1 = new Account();
Account acc2 = new Account();
DebitCard aCard = new DebitCard(acc1);  // card → acc1
acc2.setCard(aCard);                          // acc2 → card

// PROBLEM: card.theAccount = acc1, but acc2 thinks it owns card!
// Links inconsistent → violates integrity

✅ Solution — Delegate One Class to Maintain Integrity

public class Account {
  private DebitCard theCard;

  public void addCard() {
    // Account handles BOTH creation and reference setting
    theCard = new DebitCard(this);
  }
}

ArrayList / collection للـ multiple references. الـ exam Q1.07 الـ Code (a).

private Class field;

0..1
One reference, can be null
private Class field;
*
0 to many
private List<Class> items;
= new ArrayList<>();
1..*
1 to many
private List<Class> items;
(check size ≥ 1)
0..8
up to 8
List + size check < 8
💻 Q1.07 Style — Advertiser ↔ Account (1:*)

الـ UML: Advertiser [1] ━━ [*] Account

الكود الصح (هو Code A في exam):

Correct implementation
public class Advertiser {
  private Set accounts;

  public Advertiser() {
    accounts = new HashSet();
  }

  public void addAccount(Account a) {
    if (a != null) {
      accounts.add(a);
      a.setOwner(this);   // Maintain bidirectional
    }
  }
}

public class Account {
  private Advertiser owner;  // single ref — many accounts share owner

  public void setOwner(Advertiser newOwner) {
    if (newOwner != null)
      owner.addAccount(this);
  }
}
⚠️ Common Wrong Versions
  • Code B: Advertiser has single Account account field — wrong, should be Set.
  • Code C: Account has Set of Advertiser owners — wrong, that would be many-to-many, not 1-to-many.
⛓️ Many-to-Many — Two strategies
  1. Direct — Both classes hold collections. Choose one class to maintain referential integrity.
  2. Reify the association — Create an intermediate class (e.g., Signatory) holding the relationship. Often cleaner.
08

Qualified Associations & Association Classes

HashMap للـ qualifiers. Reification للـ association classes.

🔑 Qualified Association → HashMap

UML: Bank [accno] → Account [0..1] — efficient lookup by accno.

public class Bank {
  private HashMap<Integer, Account> accounts;

  public Account lookupAccount(int number) {
    return accounts.get(number);
  }

  public void addAccount(Account a) {
    accounts.put(a.getNumber(), a);
  }
}
📎 Association Class → 2 Associations + Reified Class

UML: Module *──*── Student + attached Registration {mark}

Implementation: Transform into two associations through a new class.

class Registration {
  private Student student;
  private int mark;

  Registration(Student st) {
    student = st;
    mark = 0;
  }
}

public class Module {
  private List<Registration> registrations;

  public void enrol(Student st) {
    registrations.add(new Registration(st));
  }
}
09

State Diagram → Code

3 steps: define states · track current state · switch on state per operation.

🔢 Implementation Steps
  1. Define States — enumerate as constants (or use State pattern).
  2. Track Current State — initial in constructor.
  3. Operation-by-Operation — switch with case per state.
💻 Example — Account class

States: InCredit, OverDrawn, Suspended. Operations: deposit, withdraw, suspend, unsuspend.

public class Account {
  // Step 1: define states
  private final int InCredit = 0;
  private final int Overdrawn = 1;
  private final int Suspended = 2;

  private int state;
  private int historyState;
  private double bal;

  // Step 2: initial state in constructor
  public Account() {
    state = InCredit;
  }

  // Step 3: switch on state per operation
  public void withdraw(double amt) {
    switch (state) {
      case InCredit:
        if (amt > bal) state = Overdrawn;
        bal -= amt;
        break;
      case Overdrawn:
      case Suspended:
        break;
    }
  }
}
⚡ Entry / Exit Actions

Entry action تتنفذ لما تدخل state. Exit action لما تخرج.

public void eventF() {  // transition INTO state S
  switch (state) {
    case T:
      doEntry();  // state T → S: trigger S's entry action
      state = S;
      break;
  }
}

public void eventG() {  // transition OUT of state S
  switch (state) {
    case S:
      doExit();   // state S → U: trigger S's exit action
      state = U;
      break;
  }
}

Question 1 Fall 2021–22 / Winter 2021 · Q1.07
Class diagram: Advertiser [1] ━━ [*] Account. Which code properly implements it?

(a) Advertiser has Set<Account> accounts. Account has single Advertiser owner. addAccount checks null + sets owner.
(b) Advertiser has single Account account field.
(c) Advertiser has Set; Account has Set<Advertiser> owners.
A Code (a)
B Code (b)
C Code (c)
✅ الإجابة الصح: A — Code (a) الـ multiplicity 1:* بتقول: Advertiser واحد ↔ Accounts كتيرة. Code (a) صح:
  • Advertiser → Set<Account> accounts (collection).
  • Account → Advertiser owner (single reference).
  • addAccount + setOwner للحفاظ على bidirectional integrity.
B — single field غلط، Advertiser المفروض يحمل multiple. · C — Account has Set of Advertisers = many-to-many، مش 1-to-*.
Question 2 Fall 2021–22 / Winter 2021 · Q1.08
Which class diagram represents this code?

class Order { OrderStatus status; ArrayList<OrderDetail> details; ... }
class OrderDetail { Product product; }
class Product { String code, name; float unitPrice; }
enum OrderStatus { OPEN, CLOSED; }
A Order ◆[1:*] OrderDetail → Product (1:1); Order → OrderStatus enum
B Order [1:1] OrderDetail [1:1] Product
C Order → OrderDetail (plain association); → Product
D Order [*:*] OrderDetail many-to-many
✅ الإجابة الصح: A ArrayList<OrderDetail> = Order has many OrderDetails (multiplicity *). Order owns OrderDetails فا composition ◆. OrderDetail has 1 Product. Order has OrderStatus enum.
Question 3 Practice — operation contracts
What does an operation contract consist of?
A Just the method signature
B Preconditions + Postconditions
C Only the method body
D Class name + parent class
✅ الإجابة الصح: B — Preconditions + Postconditions الـ contract بيوصف functionality formally: ما هو لازم يكون true قبل (precondition)، وما هو الـ effect (postcondition).
Question 4 Practice — multiplicity
UML: Manager [1] ━→ [*] Account. Best Java implementation?
A private Account account;
B private Manager mgr;
C private ArrayList<Account> accounts;
D private HashMap<String, Account> index;
✅ الإجابة الصح: C — ArrayList Multiplicity * = many → collection (ArrayList/Set).
A — single ref, doesn't model "many". · D — HashMap للـ qualified associations فقط.
Question 5 Practice — qualified
A qualified association in UML (e.g., Bank [accno] → Account) is best implemented with:
A ArrayList
B HashMap / HashTable
C Single reference field
D Stack
✅ الإجابة الصح: B — HashMap الـ qualifier للـ efficient lookup. HashMap بيعطي O(1) lookup بالـ key.
Question 6 Practice — state diagram
What is the recommended way to implement a state diagram in code?
A One method for each transition
B Boolean field for each state
C Inheritance from a State class
D Define states as constants and use switch in each operation
✅ الإجابة الصح: D Steps: (1) constants for states, (2) initial state in constructor, (3) switch by state in each operation.

(C) ممكن يُطبق بـ State Pattern، أنظف لكن أعقد. الـ doctor بيوصي بـ (D) للبساطة.

Question 7 Practice — order of impl
If you implement classes Top-Down, what is the main disadvantage?
A You need stubs for lower-level classes
B The system isn't testable until everything is done
C Higher classes are hard to design
D You lose object-orientation
✅ الإجابة الصح: A — Stubs needed Top-Down يبدأ بـ high-level. الـ high-level methods بتعتمد على lower classes → محتاج temporary stubs لتجربتها.
B — actually advantage of top-down: early validation. · D — مش متعلق.
Question 8 Practice — bidirectional
In a bidirectional 1:1 association between Account and Guarantor (both compulsory), what is the recommended creation order?
A Both at the same time using new (impossible in Java/C++)
B Random — order doesn't matter
C Create one with default constructor, then create second pointing to it, then set reverse link
D Make them immutable singletons
✅ الإجابة الصح: C Java doesn't allow simultaneous creation. لازم تخالف الـ constraint مؤقتاً: ابدأ بـ default Guarantor, then Account(g), then g.setAccount(a) — finalize bidirectional link.
