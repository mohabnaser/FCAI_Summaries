Introduction to Testing

"Program testing can show the presence of bugs, but never their absence" — Dijkstra.

📖 Definitions
  • Software Testing = process of executing a program with the intent of finding errors.
  • Process of establishing confidence that a program does what it's supposed to do via running it on different test data.
  • QA activity that aims to find bugs in working code and fix them.
👥 Tester vs QA Engineer
  • Tester: develops test cases and runs programs to find bugs.
  • QA Engineer: plans QA activities; creates and enforces standards/methods to prevent bugs from occurring.
💡 Lion King 1994 DisasterDisney released "Lion King Animated Storybook" CD-ROM without testing on different PC models. After Christmas Day 1995, endless angry parent calls. Disney had to hire support, run another debug cycle, and ship thousands of replacement CDs.

Bugs — Cost & Types

Cost of bug grows exponentially as it's found later.

💰 The Cost of Bugs (Ron Patton)
PhaseCost to Fix
Specification$1
Design$10
Code$100
Test$1,000
Release$1,000+

Goal: Find bugs as early as possible.

📜 5 Rules That Define a Bug

A bug occurs when one or more of these is true:

  1. Rule 1: The software doesn't do something that the spec says it should.
  2. Rule 2: The software does something that the spec says it shouldn't.
  3. Rule 3: The software does something that the spec doesn't mention.
  4. Rule 4: The software doesn't do something the spec doesn't mention but should.
  5. Rule 5: The software is difficult to understand, hard to use, slow ... not right.
🚨 Severity (How Bad)
1

2

Wrong result, loss of functionality
3
Minor

4

Improvement idea
⏰ Priority (How Soon)
  1. Immediate fix — blocks further testing, very obvious
  2. Must fix before release
  3. Should fix when time permits
  4. Would like to fix but can release as-is
💡 Sev vs PriSeverity = impact. Priority = urgency. Misspelling causing thousands of support calls = Severity 3, Priority 2 (annoying but not critical, fix soon).

Black-Box vs White-Box

الـ exam Q1.04 Winter 2025 سأل عن Black-box equivalence partitioning.

⬛ Black-Box (Specification-based)
  • Checks conformance with specification → function coverage.
  • Abstraction from details: source code not needed.
  • Scales up: works at unit, integration, system, acceptance levels.
  • Reveals missing functionality (which white-box can't).
  • Technique: Equivalence Partitioning + Boundary Value Analysis.
⬜ White-Box (Structural)
  • Tests against implementation.
  • Tester confident about code coverage.
  • Based on control and data flow — easier debugging.
  • Does NOT scale up — most useful at unit + integration.
  • Coverage types: Statement, Branch, Path.
  • Reveals unexpected functionality.
💡 Best PracticeUSE BOTH! They reveal different defects.
📊 Equivalence Partitioning (BB)

Partition each input into 2 or more groups:

  • Valid equivalence classes
  • Invalid equivalence classes

Test one value from each class as representative.

Example — Reward Points (Winter 2025 Q1.04)

Function returns reward points. If purchase ≤ 1000 LE → 10%. If > 1000 LE → 20%. If input is negative → throw exception.

Partitions:

  • Input ≤ 0 → exception
  • Input (0, 1000] → 10% reward
  • Input > 1000 → 20% reward

Test cases to cover all partitions:

TC1: input 100,  output 10     (10% partition)
TC2: input 1000, output 100    (10% partition — boundary)
TC3: input -100, throw exception (invalid)
// But this misses the 20% partition!

// Correct set:
TC1: 10 & M, 18 & F, 75 & F     // (Winter 2025 example)
TC2: 100 → 10,  2000 → 400,  -200 → throw
🎯 Boundary Value Analysis

Bugs are common at boundaries. Test values at, just below, and just above the boundary.

Example for age range 18-65:

  • 17, 18, 19 (lower boundary)
  • 64, 65, 66 (upper boundary)
  • -1, 0, 1 (special edge — invalid below)

Test-Driven Development (TDD) ⭐

Winter 2025 Q1.01 سأل عن TDD steps order.

🔄 TDD Definition

TDD = software development practice that focuses on creating unit test cases BEFORE developing the actual code.

It's an iterative approach combining: programming + unit tests + refactoring.

🔴🟢🔵 The TDD Cycle
🔴
1. RED
Write failing test
→
🟢
2. GREEN
Write minimal code to pass
→
🔵
3. REFACTOR
Improve without breaking
📜 3 Laws of TDD (Uncle Bob)
  1. You may NOT write production code until you have written a failing test.
  2. You may NOT write more of a test than is necessary to fail.
  3. You may NOT write more production code than is necessary to pass the currently failing test.
📋 Step Order ⭐ (Winter 2025 Q1.01)

Given these 5 steps, the correct order is:

  1. Write a test (step 2 of the original list)
  2. Test fails (step 3)
  3. Write code (step 5)
  4. Test passes (step 4)
  5. Refactor code (step 1)
🎯 Answer: 2 → 3 → 5 → 4 → 1 = (c) in the exam.
💻 Quick Example — abs() function
JavaScript — TDD cycle
// 🔴 Step 1: Write failing test
function testAbsoPos5() {
  return abso(5) === 5;
}

// 🟢 Step 2: Write minimal code
function abso(num) {
  return num;  // Just enough to pass
}

// 🔴 Step 3: Write new failing test
function testAbsoNeg5() {
  return abso(-5) === 5;
}

// 🟢 Step 4: Extend code to pass
function abso(num) {
  if (num < 0) return -num;
  return num;
}

// 🔵 Step 5: Refactor (if needed)

Python unittest Library

الـ standard library في Python للـ unit testing.

🐍 unittest Anatomy
  1. import unittest
  2. Import code to test: from my_module import my_function
  3. Create test class: class TestMyCode(unittest.TestCase)
  4. Write test methods starting with test_
import unittest
class TestSomething(unittest.TestCase):
    def test_pass(self):
        self.assertEqual(2 + 2, 4)
    def test_fail(self):
        self.assertEqual(2 + 2, 42)

if __name__ == '__main__':
    unittest.main()
✓ Common assert Methods

assertNotEqual(a, b)Passes if a != b
assertTrue(x)Passes if x is truthy
assertFalse(x)Passes if x is falsy
assertIsNone(x)Passes if x is None
assertIsNotNone(x)Passes if x is not None
assertIn(a, list)Passes if a is in list
assertAlmostEqual(a, b)For floats (handles tiny diffs)

REST APIs ⭐

Winter 2025 Q1.02: REST APIs — which is FALSE?

📡 What is a REST API?
  • API (Application Programming Interface) = set of rules to interact with another software.
  • REST (Representational State Transfer) = architectural style for web services serving data (not web pages).
  • Communicate over HTTP using standard methods.
  • Looks like a website BUT it's NOT a website. Has an endpoint.
  • Serves data, not web pages. Accessed by software.
🌐 HTTP Methods (CRUD)
GET
Read · retrieve data
POST
Create · new resource
PUT
Update existing
DELETE
Remove resource
MethodExample
GET /usersFetches list of users
POST /usersAdds a new user
PUT /users/1Updates user with ID 1
DELETE /users/1Deletes user with ID 1
🏗️ REST Architecture
  • Resources: objects/data the API manipulates (users, posts).
  • URIs (Uniform Resource Identifiers): unique endpoint per resource (/users/1).
  • HTTP Methods: GET, POST, PUT, DELETE.
  • Stateless: each request must contain ALL information needed — no session state on server. ⭐
  • Responses: status codes + data in JSON or XML.

HTTP Status Codes

CodeMeaning
200 OKSuccessful GET
201 CreatedSuccessful POST
404 Not FoundResource doesn't exist
⚠️ Winter 2025 Q1.02 — FALSE statement"They maintain state/session information from one API call to the next" is FALSE. REST is STATELESS by design — each request is independent.
💻 Python REST Example
import requests

url = "https://catfact.ninja/fact"
try:
    response = requests.get(url)
    response.raise_for_status()  # Raise on HTTP error
    data = response.json()
    print(data)
except requests.exceptions.RequestException as e:
    print(f"Error: {e}")

Question 2 Winter 2025 · Q1.02
What is FALSE about REST APIs?
A They are designed as end points to access programs via other programs
B They maintain state/session information from one API call to the next
C They have a unique URI for each endpoint
D They can be implemented via HTTP requests returning JSON or XML
✅ الإجابة الصح: B REST is STATELESS by definition. Each request must contain ALL information needed. The server does NOT keep session state.
Question 3 Winter 2025 · Q1.04
Function returns reward points: ≤1000 LE → 10%, >1000 → 20%, negative → exception. Which test cases satisfy equivalence partitioning (covering all partitions)?
A TC1: 100→10 · TC2: 1000→100 · TC3: −100→exception
B TC1: 0→0 · TC2: 100→10 · TC3: 1000→100
C TC1: 1000→100 · TC2: −200→ex · TC3: −1000→ex
D TC1: 200→20 · TC2: 2000→400 · TC3: −200→exception
✅ الإجابة الصح: D — All 3 partitions covered
  • 200 → 20 (covers 10% partition for input ≤1000)
  • 2000 → 400 (covers 20% partition for input >1000)
  • −200 → exception (covers invalid partition)
A — Only covers 10% & invalid, missing 20%. · B — No invalid partition. · C — Two negatives (same partition), no 20%.
Question 4 Winter 2025 · Q1.03
Min test cases for: (1) FULL statement coverage, (2) FULL path coverage (not branch)?
A 2 for statement, 4 for path
B 1 for statement, 2 for path
C 2 for statement, 2 for path
D 1 for statement, 4 for path
✅ الإجابة الصح: D Statement coverage = 1 TC can hit all statements. Path coverage = need 1 TC per independent path. ده classic distinction.

✅ الإجابة الصح: D — Rule 3 Adding undocumented features is a bug. The spec is the source of truth — extra functionality is uncontrolled scope.
