Corona Store — SRS

الـ business: متجر إلكتروني لبسكويت وشوكولاتة Corona.

📋 Requirements Statement
  1. Corona قررت تعمل online store للعملاء يطلبوا منتجاتها.
  2. العملاء يدفعوا بـ credit card أو cash on delivery.
  3. المهمة: بناء core business logic (UI later).
  4. النظام بيحتفظ بـ list من products. كل product: code, name, unitPrice.
  5. العميل يعمل order = list of items + quantity per product. الـ Order فيه delivery address.
  6. كل order ياخد automatic ID.
  7. الـ Order يفضل open لحد ما يتدفع، بعدها يبقى closed.
  8. Cash: العميل بيدي مبلغ معين → الـ delivery person يرجع الباقي. لو الـ cash insufficient → message + order remains open.
  9. Credit Card: ممكن تكون valid · expired · invalid · insufficient funds. لو valid → payment + user notified. غير كده → error message.

Use Case Model

الـ Actor الرئيسي = Buyer. دي الـ Use cases الأساسية.

🧍 Actors Identified
  • ✅ Buyer — main actor، بيعمل orders وبيدفع.
  • ❓ Delivery Man? — مش actor للـ system، هو موظف بيستخدم external process. إجابة الامتحان: Buyer is the only system actor.
🎬 Use Cases

Top Level

  • UC1 — Create Order
  • UC2 — Pay for Order

Specialized (Generalization)

  • UC2a — Pay by Cash (extends UC2)
  • UC2b — Pay by Credit Card (extends UC2)
💡 Use Case GeneralizationPay for Order هو الـ generic. و Pay by Cash و Pay by Credit Card هم variants. النوع ده من العلاقات بيتمثل بـ generalization arrow (open triangle) من المخصص للعام.
03

Use Case Descriptions

الـ flow التفصيلي لكل use case.

🛒 UC1 — Create Order
  • Actor: Buyer
  • Description: Buyer makes an order and decides products + quantities.

Main Flow

  1. Request to create order
  2. Give delivery address
  3. Generate automatic order ID
  4. Enter item and quantity
  5. Repeat step 4 to enter all items
  6. Create order and set status as OPEN
  7. Add to orders list

Exceptional Flow — Cancel Order

  1. Request to cancel order
  2. لو الـ order = OPEN → احذفه. غير كده → رسالة Cannot cancel.
💰 UC2 — Pay for Order (Generic)
  • Actor: Buyer
  • Description: Pays for his order.

Main Flow

  1. Request to pay order
  2. Calculate order total
  3. Provide payment (cash/credit)
  4. لو نجح الدفع → الـ order يبقى CLOSED
💵 UC2a — Pay by Cash
  1. Request to pay order
  2. Calculate order total
  3. Give cash amount
  4. لو cash >= total → take payment
  5. Order = CLOSED
  6. Return change = cash − total
  7. Display message

Exceptional Flow — Insufficient Cash

1. Display message · 2. Order remains OPEN

💳 UC2b — Pay by Credit Card
  1. Request to pay order
  2. Calculate order total
  3. Give card details
  4. Validate card with Bank
  5. Deduct amount from card
  6. لو نجح الدفع → Order = CLOSED
  7. Display message

Exceptional Flow — Card Failure

لو الكارت INVALID, EXPIRED, or INSUFFICIENT → Display message + Order remains OPEN.

Class Identification — CRC Method

Find nouns → candidate classes. حدد responsibility لكل class باستخدام CRC cards (Class-Responsibility-Collaboration).

📝 Approach
  • اقرا الـ requirements واستخرج nouns — كل noun ممكن يكون candidate class.
  • حدد responsibility لكل class.
  • استخدم CRC cards (sticky notes) لكل candidate class.
  • إذا الـ responsibilities ما تتفعش في card واحد = الـ class محتاج يتقسم.

The Full Class Diagram

اجمع كل الـ classes والـ relationships.

🗺️ Relationships Identified

1. OrderManager ↔ Order (1 : *)

OrderManager بيعمل aggregate للـ Orders (open diamond ◇). المتجر عنده many orders.

2. OrderManager ↔ Product (1 : *)

المتجر عنده catalog of products. العلاقة Aggregation.

3. Order ↔ OrderDetail (1 : *) — Composition ◆

OrderDetail ما يقدرش يوجد بدون Order (وبيتمسح معاه). دي strong composition.

4. OrderDetail ↔ Product (* : 1)

كل OrderDetail بيشاور على Product واحد. وكذا OrderDetails ممكن يشاوروا على نفس الـ Product.

5. Order ↔ Payment (1 : 1)

كل order عنده على الأكثر Payment واحد. دي plain association.

6. Payment ← Cash, Credit (Generalization)

Payment هو abstract؛ و Cash و Credit هما subclasses.

7. Order → OrderStatus (enum)

حقل الـ status في Order بيشاور على OrderStatus enum (OPEN, CLOSED).

8. Credit → CardStatus (enum)

CardStatus = VALID, EXPIRED, INVALID, INSUFFICIENT.

Sequence Diagram — Create Order

تحويل UC1 لـ sequence من method calls.

🎬 Create Order Flow
  1. Actor → OrderManager : createOrder(delivery_address)
  2. OrderManager يعمل new :Order instance ويرجع order ref
  3. Loop for each product:
    • Actor → OrderManager : addProduct(orderID, prodID, quantity)
    • OrderManager creates new OrderDetail
    • OrderManager → Order : addItem(orderDetail)
  4. OrderManager returns confirmation to Actor
Pseudo-sequence (Java method calls)
// Step 1: Create order
Order myOrder = orderManager.createOrder("Cairo, Egypt");

// Step 2: Add items in loop
for (ProductInput p : userInputs) {
  orderManager.addProduct(myOrder.getId(), p.code, p.quantity);
}

// Step 3: Pay for order
Cash payment = new Cash(100.0);
boolean success = orderManager.payForOrder(myOrder.getId(), payment);

Lessons for Question 2 ⭐

الـ template اللي تتبعه في Question 2 في الـ final.

📚 Step-by-Step Method
  1. Read SRS carefully. علم على الـ nouns والـ actions.
  2. Identify actors — مين بيتفاعل مع الـ system من برّه؟ (1 mark)
  3. Identify use cases — كل actor يقدر يعمل إيه؟ (4-6 marks)
  4. Draw use case diagram مع العلاقات (extend, include, generalization).
  5. Detail one use case — main flow + exceptional flows (4-5 marks).
  6. Identify classes من الـ nouns. صنفهم كـ Domain / Control / Boundary.
  7. Draw class diagram بالـ attributes, operations, relationships, multiplicities (8 marks).
  8. Draw sequence diagram لـ use case واحد ويوضح الـ methods المستخدمة (4 marks).
  9. Draw state diagram (لو مطلوب، 2-4 marks) لحالات object معين.
⏰ Time Allocation in Exam

Question 2 = حوالي 20+ marks. خد لها ساعة من الـ 2 ساعات.

  • 5 min — اقرأ الـ requirements وحدد الـ actors/use cases.
  • 15 min — Use case diagram + 1 detailed use case.
  • 25 min — Class diagram (الأطول).
  • 10 min — Sequence diagram.
  • 5 min — State diagram لو مطلوب.
✅ Mistakes to Avoid
  • ❌ Methods في الـ sequence diagram مش موجودة في الـ class diagram.
  • ❌ نسيان الـ multiplicities على الـ associations.
  • ❌ الخلط بين Aggregation ◇ و Composition ◆.
  • ❌ نسيان حالات OPEN/CLOSED في الـ state diagram.
  • ❌ عدم استخدام علاقة extend للـ conditional use cases (مثلاً Bad Credit).
  • ❌ رسم الـ actors كأنهم classes — الـ actors خارج الـ system boundary.
💡 Final Tip

للـ Question 2: اقرأ السؤال مرتين، وحدد الـ keywords الخاصة بالـ relationships (e.g., consists of = composition, has many = aggregation, is a kind of = generalization, uses = association). كل keyword بيدلك على relation معينة.
