Corona Store — SRS

الـ business: متجر إلكتروني لبسكويت وشوكولاتة Corona.

📋 Requirements Statement
  1. Corona قررت تعمل online store للعملاء يطلبوا منتجاتها.
  2. العملاء يدفعوا بـ credit card أو cash on delivery.
  3. المهمة: بناء core business logic (UI later).
  4. النظام بيحتفظ بـ list من products. كل product: code, name, unitPrice.
  5. العميل يعمل order = list of items + quantity per product. Order فيه delivery address.
  6. كل order ياخد automatic ID.
  7. Order open لحد ما يتدفع، بعدها closed.
  8. Cash: العميل بيدي مبلغ معين → الـ delivery person يرجع الباقي. لو الـ cash insufficient → message + order remains open.
  9. Credit Card: ممكن تكون valid · expired · invalid · insufficient funds. لو valid → payment + user notified. غير كده → error message.

Use Case Model

الـ Actor الرئيسي = Buyer. الـ Use cases الأساسية.

🧍 Actors Identified
  • ✅ Buyer — main actor، بيـ make orders ويـ pays.
  • ❓ Delivery Man? — مش actor للـ system، هو موظف بيستخدم external process. الـ exam answer: Buyer is the only system actor.
🎬 Use Cases

Top Level

  • UC1 — Create Order
  • UC2 — Pay for Order

Specialized (Generalization)

  • UC2a — Pay by Cash (extends UC2)
  • UC2b — Pay by Credit Card (extends UC2)
💡 Use Case Generalization"Pay for Order" هو الـ generic. "Pay by Cash" و "Pay by Credit Card" هم variants. النوع ده من العلاقات بيتمثل بـ generalization arrow (open triangle) من المخصص للعام.
03

Use Case Descriptions

الـ flow التفصيلي لكل use case.

🛒 UC1 — Create Order
  • Actor: Buyer
  • Description: Buyer makes an order and decides products + quantities.

Main Flow

  1. Request to create order
  2. Give delivery address
  3. Generate automatic order ID
  4. Enter item and quantity
  5. Repeat 4 to enter all items
  6. Create order and set status as OPEN
  7. Add to orders list

Exceptional Flow — Cancel Order

  1. Request to cancel order
  2. If order is OPEN → delete it. Else → "Cannot cancel" message.
💰 UC2 — Pay for Order (Generic)
  • Actor: Buyer
  • Description: Pays for his order.

Main Flow

  1. Request to pay order
  2. Calculate order total
  3. Provide payment (cash/credit)
  4. If successful → order is CLOSED
💵 UC2a — Pay by Cash
  1. Request to pay order
  2. Calculate order total
  3. Give cash amount
  4. If cash >= total → take payment
  5. Order = CLOSED
  6. Return change = cash − total
  7. Display message

Exceptional Flow — Insufficient Cash

1. Display message · 2. Order remains OPEN

💳 UC2b — Pay by Credit Card
  1. Request to pay order
  2. Calculate order total
  3. Give card details
  4. Validate card with Bank
  5. Deduct amount from card
  6. If successful → Order = CLOSED
  7. Display message

Exceptional Flow — Card Failure

If card is INVALID, EXPIRED, or INSUFFICIENT → Display message + Order remains OPEN.

Find nouns → candidate classes. Decide responsibility per class with CRC cards (Class-Responsibility-Collaboration).

📝 Approach

  • استخدم CRC cards (sticky notes) لكل candidate class.

05

The Full Class Diagram

Combine all classes + relationships.

🗺️ Relationships Identified

1. OrderManager ↔ Order (1 : *)

OrderManager aggregates Orders (open diamond ◇). The store has many orders.

2. OrderManager ↔ Product (1 : *)

Store has a catalog of products. Aggregation.

3. Order ↔ OrderDetail (1 : *) — Composition ◆

OrderDetail can't exist without Order (deleted with it). Strong composition.

4. OrderDetail ↔ Product (* : 1)

Each OrderDetail references one Product. Multiple OrderDetails can point to same Product.

5. Order ↔ Payment (1 : 1)

Each order has at most one Payment. Plain association.

6. Payment ← Cash, Credit (Generalization)

Payment is abstract; Cash and Credit are subclasses.

7. Order → OrderStatus (enum)

Order's status field references an OrderStatus enum (OPEN, CLOSED).

8. Credit → CardStatus (enum)

CardStatus = VALID, EXPIRED, INVALID, INSUFFICIENT.

Sequence Diagram — Create Order

تحويل UC1 لـ sequence من method calls.

🎬 Create Order Flow
  1. Actor → OrderManager : createOrder(delivery_address)
  2. OrderManager creates a new :Order instance (return order ref)
  3. Loop for each product:
    • Actor → OrderManager : addProduct(orderID, prodID, quantity)
    • OrderManager creates new OrderDetail
    • OrderManager → Order : addItem(orderDetail)
  4. OrderManager returns confirmation to Actor
Pseudo-sequence (Java method calls)
// Step 1: Create order
Order myOrder = orderManager.createOrder("Cairo, Egypt");

// Step 2: Add items in loop
for (ProductInput p : userInputs) {
  orderManager.addProduct(myOrder.getId(), p.code, p.quantity);
}

// Step 3: Pay for order
Cash payment = new Cash(100.0);
boolean success = orderManager.payForOrder(myOrder.getId(), payment);

Lessons for Question 2 ⭐

الـ template اللي تتبعه في Question 2 في الـ final.

📚 Step-by-Step Method
  1. Read SRS carefully. Highlight nouns and actions.
  2. Identify actors — Who interacts with the system from outside? (1 mark)
  3. Identify use cases — What can each actor do? (4-6 marks)
  4. Draw use case diagram with relationships (extend, include, generalization).
  5. Detail one use case — main flow + exceptional flows (4-5 marks).
  6. Identify classes from nouns. Categorize as Domain/Control/Boundary.
  7. Draw class diagram with attributes, operations, relationships, multiplicities (8 marks).
  8. Draw sequence diagram for one use case showing methods used (4 marks).
  9. Draw state diagram (if asked, 2-4 marks) for an object's states.
⏰ Time Allocation in Exam

Question 2 = ~20+ marks. ساعة من الـ 2 ساعات.

  • 5 min — Read requirements + identify actors/use cases.
  • 15 min — Use case diagram + 1 detailed use case.
  • 25 min — Class diagram (the longest).
  • 10 min — Sequence diagram.
  • 5 min — State diagram if asked.
✅ Mistakes to Avoid
  • ❌ Methods in sequence diagram not in class diagram.
  • ❌ Missing multiplicities on associations.
  • ❌ Confusing Aggregation ◇ vs Composition ◆.
  • ❌ Forgetting OPEN/CLOSED states in state diagram.
  • ❌ No "extend" relationship for conditional use cases (e.g., Bad Credit).
  • ❌ Drawing actors as classes — actors are outside the system boundary.
💡 Final Tip

للـ Question 2: اقرأ السؤال مرتين، حدد الـ keywords للـ relationships (e.g., "consists of" = composition, "has many" = aggregation, "is a kind of" = generalization, "uses" = association). كل keyword بيدلك على relation معينة.
