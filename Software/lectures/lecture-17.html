SCM — Software Configuration Management

Practice of identifying artifacts & handling changes systematically.

📖 SCM Definition

SCM = practice of identifying project artifacts and handling changes systematically so that a system can maintain its integrity over time.

Includes techniques for:

  • Evaluating proposed changes
  • Tracking changes
  • Keeping copies of the system as it existed at various points in time

SCM = disciplines and techniques of initiating, evaluating, and controlling change to software products during and after a project.

🛠️ SCM Has 2 Main Activities
  • Change Management
  • Configuration Management
  • Sometimes also: Release Planning
❓ Typical SCM Questions
  • "What is the correct version of the software module that I have to continue its coding?"
  • "What version of the design document matches the software version we are adapting?"
  • "What version is installed at ABC Industries?"
  • "What changes have been introduced in this version?"

Version Control Systems

Repository of files + tracking every change with who/when/what/why.

📦 What is VCS?
  • VCS = repository of files.
  • Every change is tracked along with who made it.
  • Keeps track of versions and changes:
    • When? When was the change made?
    • Who? Who made the change?
    • What? What did you change?
    • Why? (Through commit messages)
✨ Why Use VCS?
  • Reversion: revert to a known-good version if change is bad.
  • Change / Bug Tracking.
  • Branches: introduce features without messing up working code.
  • Merging branches: combine new code with good old code safely.
  • Parallel work: multiple devs on the same file.
📌 Fall 2021-22 / Winter 2021 Q1.11"Which is true about VCS?" — VCS allows tracking changes (a), retrieving older versions (d), working in parallel (c). It does NOT help discover bugs (b is FALSE).
Answer: (f) (a), (c), (d).
🏛️ Centralized vs Distributed

Centralized (SVN, CVS)

Single server stores all versions. Clients check out working copies.

Distributed (Git, Mercurial)

Each developer has a full copy of the repository. Work offline.

Change Control

How changes are formally proposed, evaluated, and approved.

📝 Change Request (CR)

A CR is a request to:

  • Expand or reduce project scope
  • Modify policies, processes, plans, or procedures
  • Modify costs or budgets
  • Revise schedules

CR can be originated by anyone at any point in the SW life cycle.

🔄 Types of Changes
  • Bug fixing (corrective action)
  • Adding features
  • Simplifying code (refactoring)
  • Changing schedule
⚙️ Change Control Process
  1. Someone submits a Change Request.
  2. CR is logged in bug tracking system.
  3. Impact analysis done — cost, schedule, technical impact.
  4. CCB (Change Control Board) reviews and decides.
  5. If approved: assigned, implemented, tested, deployed.
  6. If rejected: documented with reasoning.

Git Basics

Created by Linus Torvalds, 2005. For Linux Kernel development.

📜 Git History
  • Created by Linus Torvalds.
  • Started in 2005.
  • For managing Linux Kernel development.
  • Distributed VCS, fast, file corruption check.
🎯 Git Properties
  • Distributed version control system.
  • Tracks changes to files over time.
  • Allows collaboration on projects.
  • Fast and efficient.
  • Supports non-linear development (branching).
  • Open-source and free.

Git Workflow ⭐

Q1.14 / Q1.11 / Q1.08 في كل امتحان عن trio: add → commit → push.

🔄 The 4 Stages
1. Working Directory

Where you edit files

2. Staging Area

Snapshot to commit

git add
3. Local Repo

Where commits live

git commit
4. Remote Repo

GitHub/GitLab

git push
⌨️ Common Git Commands
CommandPurpose
git statusCheck status
git initCreates new local Git repo
git add <file>Stage file for commit
git add .Stage all changes
git commit -m "msg"Create commit with message
git commit -a -m "msg"Commit all tracked files
git clone <url>Clone existing repo
git pushPush commits to remote
git pullFetch + merge from remote
git logShow commit history
⭐ The Trio: Add File to Shared Repo

Standard sequence to add a new untracked file to a shared Git repo:

  1. git add <file> — stage the file
  2. git commit -m "message" — create commit locally
  3. git push — push to remote (shared) repo
📌 Q1.14 Fall 2021-22 / Q1.14 Winter 2021 / Q1.08 Winter 2025The sequence is add → commit → push. NOT init/commit/add (init creates a NEW repo, not for adding files to existing one).
💻 Local Repo Workflow Example
# 1. Initialize a new repo
mkdir my-project
cd my-project
git init

# 2. Add a file
echo "# My Project" > README.md
git add README.md
git status   # File now staged

# 3. Commit it
git commit -m "first commit"

# 4. First-time identity setup
git config --global user.email "you@example.com"
git config --global user.name "Your Name"

# 5. Check log
git log
git log --pretty=full

Git Branching

Separate development lines. Feature, Release, Hotfix branches.

🌿 What is a Branch?

A Branch = separate line of development within your repo. Allows working on features/experiments/bugfixes without affecting main code.

Creating & Switching

git branch <name>        # Create branch
git checkout <name>      # Switch to it

# Or combined:
git checkout -b <name>   # Create + switch

# List branches
git branch -l
🔀 Branching Strategies
  • Feature Branches: developing new features.
  • Release Branches: preparing a release.
  • Hotfix Branches: urgent bug fixes.
🔗 Merging Branches
# Switch to target branch (e.g. master/main)
git checkout master

# Merge source branch into target
git merge first-branch

# Verify with log
git log

# Delete merged branch
git branch -d first-branch

Question 1 Fall 2021–22 / Winter 2021 · Q1.11
Which of the following is true about VCS?
(a) They allow tracking changes in the code
(b) They helps discover bugs in the code
(c) They allow developers to work in parallel, even on the same file
(d) They allow retrieving an older version of the code
A (a) only
B (b) only
C (c) only
D (d) only
E (a), (b), and (d)
F (a), (c), and (d)
✅ الإجابة الصح: F — (a), (c), (d)
  • (a) ✓ — VCS tracks every change with who/when.
  • (b) ❌ — VCS doesn't discover bugs by itself.
  • (c) ✓ — Multiple devs can work in parallel via branches.
  • (d) ✓ — Revert to any older version any time.

Question 7 Winter 2025 · Q1.09
Version numbering: major.minor.milestone.revision-build. Milestone: 0=pre-alpha, 1=alpha, 2=beta, 3=RC, 4=final. Which is the fifth bug fix of the beta version of v2.3?
A 2.3.5.2-121
B 2.3.beta.5-132
C 2.3.2.5-33
D 2.3.2-5
✅ الإجابة الصح: C — 2.3.2.5-33

Question 8 Winter 2025 · Q1.10
Which of these questions, a version control system CANNOT answer?
A Who changed this source code file?
B Why was this file changed?
C What changed in this source code file?
D When was this file changed?
✅ الإجابة الصح: B — Why VCS records who, what, when. The "why" depends on the quality of commit messages — VCS itself can't enforce/guarantee it. (Bonus tip: write meaningful commit messages!)
