What Happens When You Open a Website? — إيه اللي بيحصل لما بتفتح موقع؟

رحلة الـ Request من البداية لحد النهاية

🔄 دورة حياة الـ Web Request

كل مرة بتضغط على لينك أو بتكتب URL في المتصفح، بيحصل تسلسل محدد من الخطوات. ده اللي بنسميه الـ Request-Response Cycle.

المستخدم بيكتب URL أو بيضغط على لينك أو بيسبمت فورم

السيرفر بيبعت HTTP Response فيه Status Code والمحتوى (HTML أو JSON)

02

Client & Server — طرفا المحادثة

مين بيطلب ومين بيرد؟

💻 Client — العميل
  • عادةً المتصفح (Browser) أو تطبيق الموبايل
  • هو اللي بيبدأ المحادثة بإرسال الـ Request
  • وظيفته يعرض HTML و CSS و JavaScript للمستخدم
  • يجمع Input من الأزرار والـ Forms ويبعتها
  • يستقبل الـ Response ويعرضها (Rendering)
🖥️ Server — الخادم
  • جهاز أو خدمة بتسمع وبترد على الـ Requests
  • هو اللي بيستقبل ويعالج الـ Request
  • بيشغل الـ Business Logic والـ Validation
  • بيتواصل مع الـ Database لو محتاج
  • بيرجع Response مناسبة (صفحة، JSON، Redirect)
💡 من المحاضرة: "The client asks; the server answers." — دي أبسط طريقة تتذكر بيها العلاقة بينهم.
03

Frontend vs Backend vs Database — الطبقات الثلاث

كل طبقة عندها مسؤولية مختلفة — تفهمهم كويس مهم جداً!

🖥️ Frontend (Client Side)

  • اللي المستخدم بيشوفه ويتعامل معاه
  • HTML, CSS, JavaScript
  • بيشتغل في المتصفح (Browser)
  • بيهتم بالشكل والتفاعل (UX)
  • بيبعت Requests للـ Backend

⚙️ Backend (Server Side)

  • بيعالج الـ Requests القادمة من المستخدم
  • Python, JavaScript, Java, PHP…
  • بيشتغل على الـ Server
  • Business Logic, Validation, Permissions
  • بيتواصل مع الـ Database ويجهز الرد

🗄️ Database (Data Layer)

  • بيحفظ البيانات بشكل دائم (Persistent)
  • SQLite, MySQL, PostgreSQL
  • لا تتكلم مع المستخدم مباشرة!
  • الـ Backend هو الوسيط الوحيد
  • Tables: students, courses, grades…

Static vs Dynamic Websites — ثابت أم ديناميكي؟

ليه Backend موجود أصلاً؟

📄 Static Website — موقع ثابت

  • نفس المحتوى لكل الزوار بدون تغيير
  • ملفات HTML جاهزة على السيرفر
  • مناسب للمواقع التعريفية البسيطة
  • عشان تعدل لازم تعدل الـ HTML يدوياً
  • لا يحتاج Database

⚡ Dynamic Website — موقع ديناميكي

  • المحتوى بيتغير بناءً على المستخدم أو الداتا
  • الصفحة بتتولد لحظياً بكود الـ Backend
  • ضروري للمتاجر، البوابات، الـ Dashboards
  • البيانات بتيجي من الـ Database
  • يدعم Login وبيانات مخصصة لكل شخص
⚠️ أمثلة على المواقع الديناميكية: البوابة الجامعية (portal.cu.edu.eg) — بتشوف درجاتك أنت بالتحديد، مش درجات كل الطلاب. ده بيحصل بسبب الـ Backend والـ Database.

URL Anatomy — تشريح الرابط

كل جزء في الـ URL بيقول حاجة مهمة

🔍 ما هو الـ URL؟

الـ URL (Uniform Resource Locator) هو عنوان أي مورد على الويب. بيقول للمتصفح فين يروح وإيه اللي يطلبه. ممكن يشير لصفحة، صورة، API endpoint، أو ملف.

HTTP — لغة التخاطب بين المتصفح والسيرفر

HyperText Transfer Protocol — البروتوكول الأساسي للويب

💬 ما هو الـ HTTP؟

الـ HTTP (HyperText Transfer Protocol) هو البروتوكول اللي بيتحكم في شكل ومضمون الـ Requests والـ Responses بين المتصفح والسيرفر. هو مش بيحدد الـ Business Logic، هو بس بيحدد شكل التواصل (Communication Format).

الـ HTTPS هو نفس الـ HTTP لكن مشفّر (Encrypted) عن طريق SSL/TLS، وبيشتغل على الـ Port 443 بدل 80.

06A

Request & Response — بنية الرسالتين

كل محادثة على الويب فيها Request من الـ Client وResponse من الـ Server

📤 HTTP Request — الطلب
  • يحتوي على Request Line (Method + Path + Version)
  • يحتوي على Headers (بيانات وصفية)
  • ممكن يحتوي على Body (بيانات مرسلة مثل الفورم)
  • الـ Body مش إلزامي — GET مثلاً مبيكونش فيه Body
HTTP REQUEST — مثال
GET /students/list HTTP/1.1
Host: portal.cu.edu.eg
Accept: text/html
Authorization: Bearer token123
(Body فاضي في الـ GET)
📥 HTTP Response — الرد
  • يحتوي على Status Line (Version + Status Code + Message)
  • يحتوي على Headers (بيانات وصفية عن الرد)
  • يحتوي على Body (HTML, JSON, صورة، أو رسالة خطأ)
  • الـ Status Code مهم جداً — بيقول إيه اللي حصل
HTTP RESPONSE — مثال
HTTP/1.1 200 OK
Content-Type: application/json
Content-Length: 256

{"status": "success", "data": [...]}
💡 القاعدة الذهبية — سؤال متكرر جداً: الـ HTTP Request لازم دايماً يحتوي على: Request Line + Headers. أما الـ Body فهو اختياري وبيكون موجود في POST/PUT بس — مش في GET. السؤال اللي بيجي في الامتحان: "An HTTP request message always contains ___" — الإجابة: A Request Line and A Header.
💡 Request Line vs Status Line — فرق مهم:
  • أول سطر في الـ Request = Request Line (Method + URL + Version)
  • أول سطر في الـ Response = Status Line (Version + Code + Message)
📋 Headers والـ Body

Headers — بيانات وصفية (Metadata):

  • Content-Type — نوع المحتوى (HTML, JSON, image)
  • Authorization — بيانات الـ Authentication
  • Content-Length — حجم المحتوى
  • Expires — تاريخ انتهاء الـ Cache

Body — المحتوى الفعلي:

  • في الـ Request: بيانات الفورم أو JSON
  • في الـ Response: HTML أو JSON أو صورة
  • مش كل Request بيحتاج Body
  • معظم Responses بيكون فيها Body
⭐ سؤال امتحان 2024 متكرر: "Expires: Tuesday, 17 Dec 2020 18:30 GMT" — ده Header بييجي في الـ Response (مش الـ Request) وبيعمل Time-Based Caching. الإجابة: response, time-based caching.
07

HTTP Methods — أوامر الطلب

الـ Method بيقول للسيرفر نية الـ Request — إيه اللي عايزه يعمله؟

📖 GET — القراءة
  • بيُستخدم لجلب صفحة أو بيانات (Read-Only)
  • البيانات بتظهر في الـ URL (Query String)
  • لا يحتوي على Body
  • مثال: فتح صفحة قائمة الطلاب
  • Safe — مش المفروض يغير حاجة على السيرفر
📝 POST — الإرسال والحفظ
  • بيُستخدم لإرسال بيانات جديدة للسيرفر
  • البيانات بتتبعت في الـ Body (مخفية)
  • مثال: Login، Registration، إضافة ريكورد جديد
  • أكثر أماناً من GET للبيانات الحساسة
  • بيُنشئ Record جديد أو بيُنفذ Action
✏️ PUT & PATCH — التعديل
  • PUT: تعديل كامل للـ Resource (Full Update)
  • PATCH: تعديل جزئي للـ Resource (Partial Update)
  • مثال: تحديث بيانات الطالب
  • الفرق المهم: PUT = الكل، PATCH = جزء
🚨 فخ امتحان 2025: "We use the PUT method for doing partial updates." — هذا خطأ! False. الـ PATCH هو اللي بيعمل Partial Update!
🗑️ DELETE — الحذف
  • بيُستخدم لحذف Resource من السيرفر
  • لازم يكون محمي بـ Authorization
  • Idempotent — تكراره مش بيعمل مشاكل إضافية
  • مثال: حذف كورس أو ريكورد
⚖️ مقارنة GET vs POST — سؤال الامتحان المتكرر
المعيارGETPOST
الغرضقراءة البياناتإرسال/حفظ بيانات جديدة
مكان البياناتفي الـ URL (مرئية!)في الـ Body (مخفية)
الأمانأقل أماناً للبيانات الحساسةأكثر أماناً
Bodyلا يحتوي على Bodyيحتوي على Body
الاستخدامفتح صفحات، بحثLogin، Registration، Forms
🚨 سؤال امتحان 2021 متكرر: "______ is more robust and secure than ______" — الإجابة: POST, GET. الـ POST أأمن لأن البيانات بتتبعت في الـ Body وملهاش ظهور في الـ URL.

🤔 401 vs 403 vs 404 vs 500 — الفروق المهمة
🚨 فروق مهمة جداً في الامتحانات:
  • 401 Unauthorized: إنت مش عامل Login أصلاً — "مين إنت؟" — السيرفر شغال ومتاح
  • 403 Forbidden: إنت عامل Login لكن معكش صلاحية — "أنا عارفك بس مش هسمحلك" — السيرفر شغال ومتاح
  • 404 Not Found: المسار أو الـ Resource غلط أو مش موجود
  • 304 Not Found: ❌ غلط! 304 = Not Modified (Cache لسه صالح) — Not Found دي 404
  • 500 Internal Server Error: الكود ضرب جوه السيرفر — المسار صح لكن في Bug

Validation & Forms — التحقق والفورمات

ليه Backend Validation ضروري حتى لو الـ Frontend بيفحص؟

🖥️ Frontend Validation
  • بيتم في المتصفح قبل إرسال الطلب
  • هدفه تحسين تجربة المستخدم (UX)
  • سريع — مش بيحتاج رحلة للسيرفر
  • يمكن تجاوزه! المستخدم يقدر يعدله
  • JavaScript, HTML5 attributes
🔐 Backend Validation
  • بيتم على السيرفر بعد استقبال الطلب
  • هدفه حماية البيانات والأمان
  • لازم يفحص الـ required fields والـ formats
  • لا يمكن تجاوزه — هو الحكم الأخير
  • ضروري حتى لو الـ Frontend فاحص
🚨 قاعدة الأمان: "Never trust input just because it came from a nice form." الـ Frontend Validation ممكن يتجاوزه أي مستخدم بسهولة. الـ Backend Validation هو الحماية الحقيقية.
🔑 مثال عملي — Login Form Flow
1
المستخدم بيملأ الفورم
Email + Password في صفحة الـ Login
2
Browser بيبعت POST Request
البيانات في الـ Body — مش في الـ URL
3
Server بيعمل Validation
بيتحقق من الـ Email والـ Password في الـ Database
4
السيرفر بيرد
لو نجح: Session + Redirect (302). لو فشل: Form مع Error Message.

JSON & APIs — نقل البيانات الحديث

ازاي الـ Backend بيبعت الداتا للـ Frontend والتطبيقات

📦 ما هو الـ JSON؟

الـ JSON (JavaScript Object Notation) هو format نصي لتمثيل البيانات المنظمة. شائع جداً في الـ APIs والتطبيقات الحديثة. بيستخدم Key-Value Pairs وشبه بالـ Dictionary في Python.

JSON — مثال بسيط
{
    "name": "Ali",
    "level": 2,
    "department": "CS",
    "courses": ["IS231", "CS101"]
}
🔗 HTML Response vs JSON Response

HTML Response — رد عادي للمتصفح:

HTML RESPONSE
HTTP/1.1 200 OK
Content-Type: text/html

<html><body>Welcome!</body></html>

JSON Response — رد الـ APIs الحديثة:

JSON API RESPONSE
HTTP/1.1 200 OK
Content-Type: application/json

{"status": "success"}
💡 سؤال امتحان 2025: "An HTTP response content type denotes ___" — الإجابة: The media type of the content. الـ Content-Type بيحدد نوع الداتا (HTML, JSON, image) — وملوش علاقة بالحجم.
🌐 ما هو الـ API؟

الـ API (Application Programming Interface) هو عقد للتواصل مع برنامج عن طريق Requests والـ Responses. الـ Web API بيرجع Responses فيها Data بدل صفحات HTML كاملة — ونفس مبادئ HTTP لازم تنطبق: URL, Method, Headers, Body, Response.

11

Cookies & Sessions — ازاي السيرفر بيتذكرك

بعد الـ Login، السيرفر عارفك إزاي في الطلبات الجاية؟

🍪 Cookie
  • بيانات صغيرة مخزنة في المتصفح
  • السيرفر بيبعتها في الـ Response
  • المتصفح بيبعتها مع كل Request تاني
  • مثال: بيانات الـ Login Session
  • الهيدر: Expires بيحدد مدة صلاحيتها
🗝️ Session
  • بيانات مخزنة على السيرفر
  • بتمثل حالة المستخدم (Logged In State)
  • السيرفر بيتعرف على المستخدم عن طريق Session ID
  • الـ Session ID بيتخزن في Cookie في المتصفح
  • Django بيتعامل معاها تلقائياً
💡 مقدمة فقط: الـ Cookies والـ Sessions موضوع تفصيلي بيجي في محاضرات الـ Django. المهم دلوقتي إنك تفهم المفهوم الأساسي: السيرفر بيستخدمهم عشان "يتذكر" المستخدم بين الـ Requests المختلفة.

Q3 Final 2021 — Q.5
The first line in the request message is called the ______ line, and the first line in the response message is called the ______ line.
A
Request, Status
B
Status, Status
C
Response, Status
D
None of The Above
✅ الإجابة الصح: A — Request, Status

Q6 Final 2021 — Q.3
Find the oddly matched HTTP status codes:
A
200 — OK
B
400 — Bad Request
C
301 — Moved Permanently
D
304 — Not Found
✅ الإجابة الصح: D — 304 — Not Found
