Authentication

Django's built-in user authentication system

🔑 Login & Logout
from django.contrib.auth import authenticate, login, logout

def login_view(request):
    if request.method == 'POST':
        username = request.POST['username']
        password = request.POST['password']
        user = authenticate(request, username=username, password=password)
        if user is not None:
            login(request, user)         # creates session
            return redirect('dashboard')   # 302 redirect
        else:
            # wrong credentials → show error
            pass

def logout_view(request):
    logout(request)   # destroys session
    return redirect('login')
Flow: authenticate() checks credentials → login() creates session → redirect with 302. Failed login → show form again with error message.
🛡️ Protecting Views
from django.contrib.auth.decorators import login_required

@login_required           # redirects to login if not authenticated
def dashboard(request):
    return render(request, 'dashboard.html')
Security: Always use @login_required on views that show sensitive data. Without it, anyone can access the view directly by URL — a 403 vs 401 scenario.

Sessions & Cookies

Maintaining state across HTTP requests

💾 Working with Sessions
# Set a session variable
request.session['user_id'] = 42

# Get a session variable
user_id = request.session.get('user_id', None)

# Delete a session variable
del request.session['user_id']

# Clear all session data (on logout)
request.session.flush()
How it works: Django stores session data on the server (in DB/cache/file). The browser only gets a session ID cookie. Cookies are created by the server and sent via Set-Cookie header.

Permissions & Authorization

What users are allowed to do

🛡️ Permission Checking
from django.contrib.auth.decorators import permission_required

@permission_required('app.can_publish')  # → 403 if no permission
def publish_article(request):
    pass

# Manual check in view
if request.user.has_perm('app.can_edit'):
    # user is authorized
    pass
else:
    return HttpResponseForbidden()   # → 403
401 vs 403:
401 = Not authenticated (need to login first)
403 = Authenticated but NOT authorized (logged in but no permission)

Middleware

Processing requests and responses globally

🔄 What is Middleware?
  • Middleware sits between the web server and Django views.
  • Processes every request before it reaches the view, and every response after leaving the view.
  • Common uses: authentication, session handling, CSRF protection, security headers.
Key built-in middleware: SecurityMiddleware, SessionMiddleware, AuthenticationMiddleware, CsrfViewMiddleware.

Django Part IV — Exam Quiz

Auth, sessions, permissions — IS231

Q1
After a successful login in Django, the typical HTTP response is:
A
200 OK with the dashboard HTML
B
401 Unauthorized
C
302 Redirect to dashboard
D
201 Created
✅ Correct: C — 302 Redirect
The Post/Redirect/Get pattern: after a successful POST (login), Django redirects to the dashboard with a 302 to prevent form re-submission on refresh.
Q2
The @login_required decorator returns which status when an unauthenticated user accesses a protected view?
A
302 Redirect to login page
B
401 Unauthorized
C
403 Forbidden
D
404 Not Found
✅ Correct: A — 302 Redirect to login page
Django's @login_required redirects unauthenticated users to the login page (302), appending ?next= to return them after login. It does NOT return 401.
Q3
A user is logged in but tries to access an admin-only page they don't have permission for. The server should return:
A
401 Unauthorized
B
403 Forbidden
C
404 Not Found
D
500 Internal Server Error
✅ Correct: B — 403 Forbidden
403 = Authenticated (logged in) but NOT authorized (no permission). 401 = Not authenticated at all. This is the classic 401 vs 403 distinction.
Q4
A cookie is created by the ______ and stored by the ______:
A
Server, Browser (client)
B
Browser, Server
C
Server, Server
D
Browser, Browser
✅ Correct: A — Server creates, Browser stores
The server sends Set-Cookie in the response header. The browser stores the cookie and sends it back in every subsequent request in the Cookie header.
Q5
Which Django command should you run before deploying to production to serve static files (CSS, JS)?
A
python manage.py runserver
B
python manage.py migrate
C
python manage.py collectstatic
D
python manage.py makemigrations
✅ Correct: C — collectstatic
collectstatic gathers all static files from all apps into STATIC_ROOT, ready to be served by a web server (nginx/Apache) in production. The dev server handles static files automatically but production doesn't.
